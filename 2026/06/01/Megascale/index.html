MegaScale：将大语言模型训练扩展到超过 10,000 个 GPU

文章从系统视角阐明了 LLM 训练的巨大规模引入的两个具体挑战。第一个挑战是在规模化条件下实现高训练效率，即如何提高模型利用率 MFU。第二个挑战是在规模化条件下实现高训练稳定性。

MegaScale 是一个为 LLM 训练定制的算法—系统协同设计的专用系统。应用两个系统原则：算法—系统协同设计(algorithm-system co-design)和深入可观测性(in-depth observability)。

主要工作包括对模型架构进行了若干修改，并纳入了有效的优化技术，包括并行 transformer block、滑动窗口注意力 和 LAMB 优化器；利用混合并行策略；基于每种并行策略的模式设计了定制技术，以最大化通信和计算之间的重叠；应用预取和基于树的加载来优化数据流水线；利用非阻塞异步操作，并消除用于大规模集合通信组初始化的全局屏障；设计了一种定制网络拓扑，减少 ECMP 哈希冲突，定制拥塞控制，并调优重传超时参数以获得高网络性能。

针对稳定性问题，应用深入可观测性的原则来构建一组诊断工具。所谓“深入可观测性”，是指一种综合性的监控和可视化策略，它穿透表层指标，收集系统栈中每个组件的详细、细粒度数据，目标是创建系统性能的多维视图。这组工具允许通过揭示导致稳定性问题的复杂交互和依赖关系来诊断系统并识别根因；开发了一个鲁棒训练框架，以自动化故障定位和恢复；设计了封装多种信息形式的心跳消息，以促进实时异常检测并提供早期预警；实现了一套诊断测试，用于识别造成中断的节点；优化了检查点和恢复流程，以减少中断等。

1. 算法优化

在算法层面对模型做了一些修改，并纳入了近期优化，以提高训练效率，同时不损害准确性。采用 transformer block 的并行版本，而不是标准的串行形式。通过这种方法，attention block 和 MLP block 的计算可以并行执行，从而减少计算时间。

滑动窗口注意力（SWA）。 滑动窗口注意力是一种稀疏注意力机制，它在输入序列中的每个 token 周围使用一个固定大小的窗口。其计算复杂度为 O(s × w)，其中 s 是输入序列长度，w 是固定窗口大小。在 w ≪ s 的情况下，滑动窗口注意力比完整自注意力更高效，后者的计算复杂度为 O(s × s)。这使得训练更快，同时不损害准确性。

大规模高效训练通常会受到 batch size 约束的阻碍。特别是，增加 batch size 可能会对模型收敛产生不利影响。LAMB 优化器已被证明能够将 BERT 训练的 batch size 扩展到 64K，同时不损害准确性。在 LLM 场景下，我们的实验发现 LAMB 可以在不损失准确性的情况下将 batch size 扩展到 4×。MegaScale 也通过 LAMB 优化器减少了 87.5% 的流水线气泡。

4. 网络性能调优

数据中心网络由基于 Broadcom Tomahawk 4 芯片的高性能交换机构建。每个 Tomahawk 芯片的总带宽为 25.6Tbps，具有 64×400Gbps 端口。三层交换机以类似 CLOS 的拓扑连接，以连接超过 10,000 个 GPU。对于每一层的交换机，下行链路与上行链路之间的带宽比例为 1:1。也就是说，32 个端口用作下行链路，32 个端口用作上行链路。该网络以较小直径提供高带宽。每个节点都可以在有限跳数内与其他节点通信。

首先，在 top-of-rack（ToR）交换机层面，一个 400G 下行端口通过特定 AOC 线缆拆分为两个 200G 下行端口。由于每条上行链路的带宽是下行链路带宽的两倍，冲突概率降低。其次，服务器上的八个 200G NIC 以 multi-rail 方式连接到八个不同交换机。由同一组 ToR 交换机连接的 GPU 服务器数量可以达到 64。并且我们策略性地调度训练任务中的数据密集型节点，使其在同一 Top of Rack（ToR）交换机下运行。这种方法显著减少了通信所需的交换机跳数，并进一步降低了 ECMP 哈希冲突概率。

拥塞控制。 在分布式训练中，当在规模化条件下采用默认 DCQCN 协议时，all-to-all 通信可能导致拥塞和更高水平的 Priority Flow Control（PFC）。过度使用 PFC 可能导致 head-of-line（HoL）阻塞，从而降低网络吞吐量。为了缓解这些问题，我们开发了一种结合 Swift 和 DCQCN 原则的算法，该算法将 Round-Trip Time（RTT）的精确测量与 Explicit Congestion Notification（ECN）的快速拥塞响应能力相结合。这种方法显著提升了吞吐量，并最小化了与 PFC 相关的拥塞。

重传超时设置。 NCCL 中的参数可以被设置以控制重传计时器和重试次数。我们调优这些参数，以在链路抖动下实现快速恢复。为了进一步减少恢复时间，我们在 NIC 上启用 adap_retrans 特性。该特性支持以更短间隔进行重传，并在链路抖动周期较短时帮助更快恢复传输。

5. 容错

  1. 自动化恢复流程
    搭了一套”监工”体系:driver(总指挥)通过 Kubernetes 分配资源,每个节点配一个 executor(管家),管家上跑个 daemon 定期发心跳汇报状态。一旦 driver 发现某进程异常、或某节点心跳超时不回,就触发恢复:暂停所有训练→让各节点跑自检诊断→揪出坏节点→通知 Kubernetes 驱逐坏节点、补上等量的健康节点→从最新检查点恢复训练。也支持人工手动踢节点。

  2. 数据收集与监控
    心跳里塞了多种信息用来提前发现问题:节点基本信息、训练进程状态、实时聚合分析的 stdout/stderr 日志(命中 warning/error 关键词就报警),还有 RDMA 流量指标。
    RDMA 流量是个巧妙的招:有些异常不报错,表面看训练在正常跑,但因为训练有周期性,每步流量模式应该差不多——所以流量突然下降或异常波动就是出问题的信号,完全停了就自动触发恢复。
    还做了一套毫秒级精度的监控系统:秒级监控看整体健康(查 ECN/PFC/QoS 配置、链路抖动、网卡问题等);毫秒级监控判断网络是否拥塞、并行传输是否到了物理极限。

  3. 诊断测试
    设计原则是又轻又全:太慢会浪费训练时间,误报率高又会误踢好机器。包含两类:

主机内网络测试:Loopback 测试(测网卡到内存/GPU 的回环带宽,全互联覆盖所有链路,定位掉速链路和 PCIe 异常);RNIC-to-RNIC 测试(测同主机不同网卡间的连通性和带宽)。
NCCL 测试:节点内 GPU 间跑 all-to-all 看带宽是否达标;通过后再和同一 ToR 交换机下的邻机跑 all-reduce,测节点间 GPU 通信。

  1. 快速检查点与恢复
    用”两阶段存 + 广播读”把检查点和恢复的开销压到最低。

6. 实验与部署经验

  1. 训练性能
    对于弱扩展性：MegaScale 靠 3D 并行通信重叠做到了近似线性扩展。
    强扩展性：MegaScale 全程比 Megatron-LM 快 1.23×~1.34×。

  2. 稳定性
    真实生产运行(>1 万卡、数千亿参数、数万亿 token、跑了数周):loss 持续收敛,期间因故障重启超过 100 次,其中90% 以上的软硬件故障由容错框架自动处理,其余靠第 5 章的诊断工具解决。
