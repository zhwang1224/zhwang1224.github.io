InfiniBand 是什么？从交换式 Fabric、RDMA 到 AI/HPC 集群

在大规模 AI 训练和高性能计算集群中，成百上千个 CPU、GPU 和存储节点需要频繁交换数据。此时，单个计算节点的性能并不是唯一瓶颈：如果网络延迟过高、带宽不足，或者通信过程消耗了大量 CPU，昂贵的计算资源就会等待在数据传输上。

InfiniBand 正是为这类场景设计的高性能互连技术。InfiniBand 是一种面向服务器、加速器和存储系统的开放互连标准，它采用点到点交换式 Fabric、基于信用的链路级流控和原生 RDMA，为 HPC 与 AI 集群提供高带宽、低延迟和低 CPU 开销的数据传输。

它不是一种“更快的以太网”，也不只是某一种网卡或线缆，而是一套覆盖物理链路、交换、寻址、路由、传输、拥塞控制和网络管理的完整体系结构。InfiniBand 规范由 InfiniBand Trade Association（IBTA）维护。IBTA 对 InfiniBand 的定义同样强调，它是一种用于连接服务器、通信设备、存储和嵌入式系统的 I/O 架构。

一、为什么需要 InfiniBand？

传统 TCP/IP 网络的数据发送过程通常涉及系统调用、内核协议栈处理、数据复制、中断以及 CPU 对网络协议的参与。对于网页访问等通用业务，这套体系兼容性强、部署灵活；但在分布式训练、科学计算和并行存储中，节点之间可能持续交换大量小消息或大规模张量，软件栈带来的延迟和 CPU 开销会被不断放大。

InfiniBand 的设计目标可以概括为三点：

  1. 让数据更快到达：使用高速串行链路和专用交换网络，当前产品已经覆盖 NDR 400 Gb/s 和 XDR 800 Gb/s。
  2. 减少处理路径：通过 RDMA 和内核旁路，让网卡直接在本地与远端已注册内存之间搬运数据。
  3. 提供可预测的通信：通过基于信用的链路级流控、硬件可靠传输和集中式 Fabric 管理，减少拥塞丢包与软件协议处理带来的波动。

因此，InfiniBand 特别适合对通信延迟、有效带宽和 CPU 利用率敏感的工作负载，例如 MPI 并行程序、NCCL 集合通信、分布式 AI 训练、并行文件系统和 NVMe over Fabrics。

二、InfiniBand 不是一根线，而是一张 Fabric

理解 InfiniBand 的第一个关键，是理解 Fabric（交换结构）。

在 InfiniBand 网络中，端点不会共享一条总线，而是通过点到点链路连接到交换机，再由一个或多个交换级构成可扩展网络。典型部署会使用 Fat-tree/Clos，也可能采用 Dragonfly+、Torus 或面向 AI 集合通信优化的多平面、rail-optimized 结构。

一张 InfiniBand Fabric 主要包括以下组件：

组件 作用
HCA（Host Channel Adapter） 服务器侧的 InfiniBand 适配器，负责队列、DMA、报文封装、可靠传输和 RDMA 等功能
InfiniBand Switch 根据转发表在端口之间转发数据包
Physical Link 连接 HCA 与交换机或交换机之间的铜缆、光纤、连接器与收发模块
Subnet Manager（SM） 发现拓扑、分配地址、计算路由并配置整个子网
InfiniBand Router 在不同 InfiniBand 子网之间转发数据，实际集群中远少于交换机

HCA 不是一块只会收发以太帧的普通 NIC。它更像一个网络数据处理引擎：应用向它提交工作请求，HCA 使用 DMA 读取或写入主机内存，完成分段、排序、重传和完成通知等工作。支持 GPUDirect RDMA 时，数据还可以在 HCA 与 GPU 显存之间直接传输，避免绕行主机内存。

三、InfiniBand 的分层体系

1. 物理层：速率、通道与传输介质

物理层规定电气/光学信号、编码、链路宽度、连接器和介质等内容。常见 InfiniBand 代际及 4X 端口的标称单向速率如下：

代际 每通道标称速率 典型 4X 端口速率
SDR 2.5 Gb/s 10 Gb/s
DDR 5 Gb/s 20 Gb/s
QDR 10 Gb/s 40 Gb/s
FDR 14.0625 Gb/s 56 Gb/s
EDR 25 Gb/s 100 Gb/s
HDR 50 Gb/s 200 Gb/s
NDR 100 Gb/s 400 Gb/s
XDR 200 Gb/s 800 Gb/s

这里的端口速率通常指每个方向的标称速率；全双工链路可以同时双向传输，但不应把两个方向简单相加后称作单向带宽。编码和协议头还会使应用可获得的有效吞吐低于物理线速。

NDR 的典型 400G 端口由 4 条 100G 通道组成，也存在 2×100G 的 NDR200。NVIDIA 的NDR 设计文档说明了 NDR、NDR200、OSFP 与 QSFP112 等组合。XDR 800G 的官方光模块资料则给出 4×200G PAM4 的实现方式，而不是 8×100G。NVIDIA XDR 800G 收发模块说明对此有明确描述。

不同代际之间可以在设备支持时协商较低速率或链路宽度，但协议上的向后兼容并不等于任意线缆和模块都能混插。实际部署仍要同时检查交换机与 HCA 的固件支持、端口形态、breakout 方式、线缆介质和模块认证。

2. 链路层：本地转发与基于信用的无损流控

InfiniBand 的“无损”首先来自链路层的 Credit-Based Flow Control，基于信用的流控。

接收端根据可用缓冲空间向发送端通告信用。发送端只有在拥有足够信用时才能继续发送，因此不会因为下一跳缓冲区已满而盲目注入数据。这与普通以太网“先发送、队列溢出后丢包”的基本模型不同。

更准确地说，InfiniBand 能避免由正常拥塞和缓冲区溢出造成的丢包，并不意味着硬件故障、链路误码或配置错误永远不会导致报文错误。可靠传输还需要链路 CRC、端到端确认、序号和重传机制共同完成。

链路层还提供虚拟通道（Virtual Lane，VL）和服务等级（Service Level，SL）。数据包携带 SL，交换机根据配置将 SL 映射到 VL，从而实现流量隔离、QoS 和死锁规避。InfiniBand 可提供最多 15 条数据 VL，另有管理用途的 VL15；不同实现实际支持的数量可能更少。NVIDIA InfiniBand QoS 文档给出了 SL、VL 与仲裁机制的说明。

3. 网络层：LID、GID 与子网

在一个 InfiniBand 子网内部，端口使用 Local Identifier（LID）进行转发。交换机依据目的 LID 查询由 Subnet Manager 写入的线性转发表，将数据包送往下一跳。

InfiniBand 还使用全局唯一标识相关的地址体系：

  • GUID：设备、端口等对象的全局唯一标识；
  • LID：子网内用于快速转发的本地标识；
  • GID：128 位全局标识，用于标识跨子网可达的端口；
  • P_Key：用于分区和访问隔离，作用可类比 VLAN，但机制并不相同。

大多数单一 HPC 或 AI 集群可以运行在一个 InfiniBand 子网中。只有跨越子网边界时，才需要由 InfiniBand Router 根据全局路由信息转发。

4. 传输层：QP 与可靠通信

InfiniBand 应用通常不直接通过 TCP Socket 收发数据，而是通过 Verbs 接口操作 RDMA 资源。核心抽象是 Queue Pair（QP，队列对），它由发送队列 SQ 和接收队列 RQ 组成。应用把 Work Request 提交到队列中；HCA 执行后，把结果写入 Completion Queue（CQ），应用再读取完成事件。

常见 QP 传输类型包括：

类型 含义 特点
RC Reliable Connected 面向连接、可靠、有序，最常见；支持 Send/Receive、RDMA Read/Write 和原子操作
UC Unreliable Connected 面向连接但不保证重传，功能和使用范围较有限
UD Unreliable Datagram 无连接数据报，开销较小，支持一对多，但不保证可靠和有序

其中，“Send/Receive”和“RDMA Read/Write”是操作语义，而 RC、UC、UD 是传输服务类型，二者不是同一层面的分类。

5. 上层协议与软件生态

InfiniBand 并不要求应用全部从零编写 Verbs 程序。常见上层软件包括：

  • MPI：HPC 并行程序的消息传递接口；
  • NCCL：GPU 间 AllReduce、AllGather、ReduceScatter 等集合通信；
  • UCX、libfabric：向上提供统一通信抽象；
  • IPoIB：在 InfiniBand 上传输 IP，便于运行传统 IP 应用；
  • NFS over RDMA、NVMe-oF、并行文件系统：面向高性能存储访问。

需要区分标准能力和厂商增强能力：RDMA、QP、信用流控等属于 InfiniBand 架构的核心；SHARP 网内归约、特定自适应路由算法、SHIELD 网络自愈和部分遥测拥塞控制则是当前 NVIDIA Quantum 平台提供的增强功能，不应笼统地视为所有 InfiniBand 设备都必然具备的能力。

四、RDMA 到底如何工作？

RDMA，即 Remote Direct Memory Access，允许一台主机的网卡直接访问另一台主机已授权的内存区域。它的价值不只是“快”，而是减少了传统网络路径中的内核切换、数据复制和远端 CPU 干预。

以一次 RDMA Write 为例，其过程可以简化为：

  1. 通信双方创建 Protection Domain、Completion Queue 和 QP；
  2. 应用注册内存区域（Memory Region，MR），HCA 建立地址转换和访问权限信息；
  3. 双方通过控制通道交换 QP 信息、远端虚拟地址和访问密钥 rkey；
  4. 发送端把 RDMA Write 工作请求提交到 SQ，并通知 HCA；
  5. HCA 通过 DMA 读取本地内存，将数据分段并生成 InfiniBand 数据包；
  6. 交换机根据目的 LID 转发数据包；
  7. 远端 HCA 校验 QP、地址、rkey 和权限后，通过 DMA 将数据写入目标内存；
  8. 操作完成后，HCA 在 CQ 中生成 Completion Queue Entry（CQE）。

因此，“绕过 CPU”应理解为：连接建立、内存注册和权限交换完成后，稳定态数据搬运不需要远端 CPU 逐包参与，也不必经过传统内核网络协议栈。 本地 CPU 或 GPU 仍要发起工作请求，应用也必须预先完成资源创建、内存注册和密钥交换。NVIDIA 的RDMA 编程文档同样将 InfiniBand 概括为支持原生 RDMA、低延迟和低 CPU 开销的服务器与存储互连技术。

五、Subnet Manager

InfiniBand 把 Fabric 的控制平面与数据平面明确分离。

Subnet Manager 会扫描子网中的 HCA、交换机和端口，发现物理拓扑，为端口分配 LID，计算路径，并把转发表、分区、QoS 等配置写入设备。Subnet Administrator（SA）则向节点提供路径、服务和多播等信息查询。

完成配置后，普通数据包由 HCA 和交换机在硬件数据平面中直接转发，不会逐包经过 Subnet Manager。因此，SM 临时故障不一定会让已经建立且拓扑未变化的所有通信立刻停止；但新设备接入、链路变化、路由重算和子网恢复会受影响。生产环境通常部署一个 Master SM 和一个或多个 Standby SM，以避免管理控制点失效。

这也解释了 InfiniBand 与传统以太网运维思路的差异：以太网工程师熟悉 MAC、VLAN、IP、OSPF/BGP；InfiniBand 工程师还需要理解 LID、P_Key、SM/SA、路径记录、QP 状态以及 Fabric 诊断工具。

六、InfiniBand 与 Ethernet、RoCE 有什么区别？

InfiniBand 和 Ethernet 都能构建高速数据中心网络，但设计起点不同。

对比项 InfiniBand 传统 Ethernet/TCP RoCEv2
基础网络 InfiniBand Fabric Ethernet + IP Ethernet + IP/UDP
数据接口 原生 RDMA Verbs Socket/TCP 为主 RDMA Verbs
拥塞时的基本行为 基于信用的逐跳流控，面向无损运行 队列溢出可丢包，由 TCP 恢复 通常结合 ECN、PFC、DCQCN 等机制优化
内核旁路 原生支持 通常不具备 支持
控制与路由 SM 集中发现并下发子网路由 二层/三层分布式协议或控制器 沿用 Ethernet/IP 网络体系
生态 HPC/AI 性能突出，供应商较集中 开放、成熟、通用性强 兼顾 RDMA 与以太网生态，但调优复杂

RoCE 的全称是 RDMA over Converged Ethernet。它把 RDMA 传输能力放到以太网上：RoCEv1 工作在二层，RoCEv2 使用 UDP/IP，能够跨三层网络。它与 InfiniBand 可以使用相近的 Verbs 编程模型，但底层链路、寻址、路由、流控和网络管理仍然不同。因此，RDMA 不等于 InfiniBand，InfiniBand 也不是唯一支持 RDMA 的网络。

选择上，InfiniBand 更适合追求极低延迟、高有效带宽和成熟集合通信优化，并愿意采用相对统一软硬件栈的专用 HPC/AI 集群；Ethernet/RoCE 更适合看重多厂商生态、既有 IP 运维体系和通用业务融合的数据中心。最终选择还要结合集群规模、通信模式、人才结构、设备成本和长期运维，而不能只比较端口峰值速率。

七、InfiniBand 在 AI 集群中处于什么位置？

一套现代 GPU 集群往往不只有一种互连：

  • PCIe 连接 CPU、GPU、HCA 等本地 I/O 设备；
  • NVLink/NVSwitch 主要承担单机或高密度计算域内部的 GPU scale-up 通信；
  • InfiniBand 或高性能 Ethernet 负责跨服务器、跨机架的 scale-out 通信；
  • 普通 Ethernet 常用于登录、监控、调度和带外管理。

在分布式训练中，梯度同步和参数交换常通过 NCCL 触发 AllReduce、AllGather 或 ReduceScatter。InfiniBand 提供跨节点 RDMA，GPUDirect RDMA 进一步允许 HCA 直接访问 GPU 显存；自适应路由和拥塞控制用于缓解热点，SHARP 等网内计算技术还可把部分归约操作下沉到交换网络。网络在这里已经不只是“传输数据”，而是决定大规模并行效率的计算基础设施组成部分。

截至 2026 年，NDR 400G 已广泛用于高性能 AI/HPC 集群，XDR 800G 平台也已进入实际产品和参考架构。NVIDIA 官方资料显示，Quantum-X800 可提供每端口 800 Gb/s 的 XDR 连接，并集成自适应路由、遥测拥塞控制和 SHARP 等功能。Quantum-X800 官方介绍可作为当前产品状态的参考。

对于更远期的 GDR 1.6T，不宜把路线图当作已经交付的产品。IBTA 的公开介绍将 GDR 1.6 Tb/s 放在 2028 年前后的规划中；路线图时间和最终产品形态仍可能变化。IBTA InfiniBand Roadmap提供的是厂商中立的演进方向。

归根结底，InfiniBand 的核心价值并不只是端口达到 400G 或 800G，而是通过 交换式 Fabric + 原生 RDMA + 硬件传输卸载 + 基于信用的流控 + 集中式子网管理，让大规模并行程序能够以更低的软件开销、更稳定的延迟和更高的有效带宽交换数据。
