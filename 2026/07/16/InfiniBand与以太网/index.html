InfiniBand 与以太网：从 RDMA、无损网络到 AI/HPC 集群选型

在大规模 AI 训练和高性能计算系统中，“应该选择 InfiniBand 还是 Ethernet”几乎是网络方案设计阶段必然遇到的问题。

一种常见观点是：InfiniBand 延迟低、原生无损，适合大规模训练；Ethernet 成本低、生态开放，适合通用数据中心。这一结论方向上没有错，但过于简化，因为今天所说的“以太网”可能指三种差异很大的系统：

  1. 运行 TCP/IP 的传统 Ethernet；
  2. 运行 RoCEv2 的无损或低丢包 Ethernet；
  3. 面向 AI/HPC 重新设计传输和拥塞控制的 Ultra Ethernet 或厂商 AI Ethernet Fabric。

本文以 FiberMall 的《InfiniBand vs Ethernet》为切入点，结合 IBTA、IEEE、NVIDIA 和 Ultra Ethernet Consortium 的公开资料，建立一个更完整的技术比较框架。

IB 与以太网是什么

1. InfiniBand 是一套完整的交换式互连架构

InfiniBand 不只定义网卡或链路，而是从物理层、链路层、网络层到传输层和管理平面的一套完整体系。HCA、InfiniBand Switch、Subnet Manager 和物理链路共同组成 Fabric。

它使用分组交换，但采用面向通道的通信抽象：应用通过 Queue Pair（QP）提交 Send/Receive、RDMA Read/Write 或原子操作，HCA 负责 DMA、分段、可靠性与完成通知。

InfiniBand 的核心特征包括：

  • 原生 RDMA 与内核旁路；
  • 基于信用的逐跳流控；
  • Subnet Manager 集中发现拓扑、分配 LID 并下发转发表；
  • RC、UC、UD 等传输服务；
  • 虚拟通道、服务等级、分区与 QoS；
  • 面向 HPC/AI 的自适应路由、拥塞控制和网内计算扩展。

IBTA 对 InfiniBand 的定义强调，它是一种用于服务器、存储和通信设备的点到点交换式 I/O 架构，而不是传统局域网协议的简单变体。

2. Ethernet 首先是一套通用链路与帧交换标准

Ethernet 由 IEEE 802.3 等标准持续演进，规定 MAC、帧格式和多种物理介质。它的优势是开放、通用和高度互操作，能够承载 IP、TCP、UDP、存储和大量上层协议。

传统 Ethernet/IP 网络通常采用尽力而为模型：当交换机队列溢出时，报文可以被丢弃，再由 TCP 等上层协议重传。这种设计非常适合 Web、云服务和通用数据中心，但在同步密集型 AI/HPC 工作负载中，丢包重传和尾延迟放大会让大量计算节点一起等待。

不过，Ethernet 并不等于 TCP。通过 RoCEv2，Ethernet 同样可以提供 RDMA；通过 ECN、PFC、DCQCN、可编程拥塞控制、自适应路由和包喷洒等技术，它也能构建高性能 AI Fabric。

IEEE 已于 2024 年批准 IEEE 802.3df-2024，覆盖 400 Gb/s 和 800 Gb/s Ethernet。因此，峰值端口速率已经不是判断 InfiniBand 与 Ethernet 优劣的充分条件。

3. RoCEv2 是 RDMA 与 Ethernet 的结合

RoCE 的全称是 RDMA over Converged Ethernet，由 IBTA 制定：

  • RoCEv1 直接运行在 Ethernet 二层，不能跨越普通 IP 路由边界；
  • RoCEv2 使用 UDP/IP 封装，可以在三层网络中路由。

RoCE 复用了 RDMA 的 QP、Memory Region、Completion Queue 和 Verbs 编程模型，但底层换成 Ethernet/IP 网络。因此，InfiniBand 和 RoCE 可以向应用提供相近的通信语义，网络控制、转发与拥塞处理方式却并不相同。IBTA 的 RoCE 说明明确指出，RoCE 能让 RDMA 运行在普通二层或三层 Ethernet 网络之上。

图 1：三者可以承载相同的 AI/HPC 应用，但数据路径、传输语义和底层网络不同。

二、RDMA：InfiniBand 与 RoCE 的共同点和分界线

RDMA 允许网络适配器直接在本地与远端已注册内存之间搬运数据。连接建立、内存注册和权限交换完成后，稳定态数据传输不需要远端 CPU 逐包接收和复制，也不经过传统内核 TCP/IP 数据路径。

InfiniBand 的 RDMA

在 InfiniBand 中，RDMA 从传输层到链路层都是架构的原生组成部分。HCA 生成 InfiniBand 报文，交换机依据 LID 转发，链路信用机制抑制因下游缓冲区不足造成的丢包，RC 传输负责端到端确认、排序和重传。

RoCEv2 的 RDMA

在 RoCEv2 中，应用仍通过 RDMA Verbs 提交工作请求，但 HCA/RNIC 将 RDMA 报文封装在 UDP/IP 和 Ethernet 中。交换机按照标准 Ethernet/IP 方式转发，网络通过 ECN、PFC 和端点拥塞控制维持高吞吐和低尾延迟。

因此，InfiniBand 原生定义了 RDMA Fabric；RoCEv2 将 RDMA 传输承载在 Ethernet/IP Fabric 上。

GPUDirect RDMA 也不专属于 InfiniBand。只要 GPU、RNIC、PCIe 拓扑、驱动和通信库满足要求，RoCE 同样可以在 GPU 显存之间进行直接数据传输。两者的差别更多体现在网络侧的端到端集成、拥塞控制和运维复杂度，而不是 RoCE 缺少 GPU Direct Memory Access 能力。

三、无损网络特性

1. InfiniBand：基于信用的逐跳流控

InfiniBand 接收端根据可用缓冲空间向上游通告 credit。发送端只有在拥有足够信用时才能继续发送相应虚拟通道上的数据，因此在正常工作和正确配置下，可以避免因下一跳缓冲区溢出而丢包。

这是一种逐链路、接收端驱动的预防机制：先确认下游能接收，再发送。

但“无损”并不表示网络永远不会出现错误。链路误码、设备故障、软件缺陷和错误配置仍可能造成异常；CRC、确认、超时和重传机制仍然必要。

2. 传统 Ethernet：允许丢包，由上层恢复

普通 Ethernet 交换机可以在队列满时丢弃帧。TCP 通过确认、超时和重传恢复可靠性，并使用拥塞窗口调整发送速率。对于能够容忍抖动的长连接和通用业务，这是非常成熟的机制。

但 RDMA 的高速硬件传输希望避免频繁丢包和重传。一次拥塞引起的尾延迟可能阻塞集合通信中的整个同步阶段，因此 RoCE 网络通常需要更主动的拥塞控制。

3. RoCEv2：ECN 负责减速，PFC 负责兜底

一个常见的 RoCEv2 拥塞控制闭环如下：

  1. 交换机检测到队列增长，在 IP 头中设置 ECN 标记；
  2. 接收端 RNIC 收到被标记报文后，向发送端返回 CNP（Congestion Notification Packet）；
  3. 发送端 RNIC 根据 DCQCN 或其他算法降低发送速率；
  4. 当局部队列接近溢出时，PFC 可以暂停特定优先级的上游发送；
  5. 拥塞缓解后，发送端再逐步恢复速率。

NVIDIA 的RoCEv2 Congestion Management 文档将这一过程划分为 Congestion Point、Notification Point 和 Reaction Point。

PFC（Priority-based Flow Control，基于优先级的流量控制）以优先级而不是单条流为粒度暂停链路。它能阻止队列溢出，却可能带来 Head-of-Line Blocking、暂停传播和潜在死锁。因此，现代 RoCE 设计通常把端到端速率控制作为主要手段，把 PFC 视为接近溢出时的保护机制，而不是让网络长期依赖 PAUSE 维持稳定。IEEE 802.1Qbb定义了按流量类别运行的 Priority-based Flow Control。

图 2：InfiniBand 用 credit 约束逐跳发送；RoCEv2 主要通过 ECN/CNP 形成端到端减速闭环，并可使用 PFC 作为链路级保护。

四、控制平面与路由方式

InfiniBand：Subnet Manager 集中配置子网

InfiniBand 子网中只有一个处于 Master 状态的 Subnet Manager，通常还会部署 Standby SM。SM 扫描 Fabric，分配 LID，计算路径并将线性转发表写入交换机，还负责分区、QoS 和部分管理策略。

数据平面建立后，普通数据包由 HCA 和交换机硬件直接转发，不会逐包经过 SM。SM 的集中式设计使专用计算 Fabric 的拓扑和路由保持一致，但运维人员仍然需要配置路由算法、QoS、拥塞控制、固件和高可用，并非“安装 SM 后网络就自动运行”。

Ethernet：分布式协议与 SDN 并存

Ethernet Fabric 可以使用 MAC 学习、VLAN、LAG，也可以在三层 Leaf-Spine 中使用 BGP、ECMP、EVPN/VXLAN 或集中式 SDN 控制器。其优势是灵活、标准化程度高，并且容易与现有数据中心网络集成。

这种自由度也意味着 RoCE 的 QoS、MTU、PFC、ECN 阈值、队列映射、哈希、缓冲区和 NIC 参数必须在端到端路径上保持一致。配置不一致不一定让链路直接中断，却可能表现为难以定位的吞吐下降和尾延迟尖峰。

两者的核心区别可以概括为：

  • InfiniBand 倾向于用统一的 Fabric 管理体系配置专用互连；
  • Ethernet 倾向于在开放的二层/三层生态中组合协议和设备；
  • 前者减少选择空间并增强端到端一致性，后者扩大选择空间并提升与现有基础设施的兼容性。

五、性能指标

截至 2026 年，InfiniBand 与 Ethernet 都已有 800 Gb/s 产品或标准。NVIDIA 同时提供 Quantum-X800 InfiniBand 与 Spectrum-X800 Ethernet 平台；IEEE 802.3df-2024 也已经覆盖 800 Gb/s Ethernet。NVIDIA 的 X800 发布资料同时列出了两种 800G 平台。

因此，比较性能时至少要分开以下指标：

指标 含义 容易出现的误区
端口线速 单端口理论物理带宽 800G 不等于应用一定获得 800G
有效带宽 扣除协议开销、负载不均和拥塞后的吞吐 只测试单流无法代表 All-to-All
空载延迟 网络基本转发路径延迟 不能代表拥塞时表现
尾延迟 p99/p99.9 完成时间 平均延迟低不代表同步阶段稳定
消息速率 每秒可处理的小消息数量 大带宽不代表小消息性能高
集合通信时间 AllReduce 等操作完成时间 同时受拓扑、算法和网内计算影响
作业完成时间 训练或仿真的整体完成时间 才是最终业务指标，但难以脱离应用比较

InfiniBand 的优势通常不是“物理速率一定更高”，而是软硬件栈更一体化，信用流控、RDMA、路由和集合通信优化围绕同类工作负载协同设计，因此更容易获得稳定的有效带宽和尾延迟。

经过正确设计的 RoCEv2 Fabric 同样可以获得很高的吞吐和较低延迟，但其表现更依赖端到端配置、交换芯片能力、NIC 拥塞算法和运维经验。

六、InfiniBand 的网内计算与 Ethernet 的快速演进

当前 NVIDIA InfiniBand 平台提供 SHARP 网内归约、自适应路由、遥测拥塞控制和 SHIELD 等增强功能。SHARP 可以把部分 Reduce/AllReduce 运算下沉到交换网络，从而减少端点和链路上的重复数据传输。

需要注意，这些是具体平台能力，不应把所有厂商增强功能都等同于 InfiniBand 基础规范。比较时应确认采购设备和软件栈是否真正支持并启用了相应能力。

Ethernet 也正在从“通用网络加 RoCE”向面向 AI 的完整 Fabric 演进：

  • 高级 ECMP、自适应路由和包喷洒提高多路径利用率；
  • NIC 支持硬件 RDMA、乱序处理和可编程拥塞控制；
  • 交换机提供更强的遥测、队列管理与拥塞隔离；
  • Ultra Ethernet 定义新的传输、拥塞控制和多路径机制。

Ultra Ethernet Consortium 已于 2025 年 6 月发布 UEC Specification 1.0，其目标是在保留 Ethernet 开放生态的同时，构建面向 AI/HPC 的完整高性能通信栈。它表明未来的竞争不再是“InfiniBand 与普通 TCP Ethernet”，而是两套面向计算语义优化的高性能 Fabric 之间的竞争。

七、运维与生态：集成度和开放性的取舍

维度 InfiniBand RoCEv2 / AI Ethernet
标准与生态 IBTA 标准，当前产品生态较集中 IEEE/IETF/IBTA/UEC 等标准，多厂商生态广泛
网络管理 SM 统一发现和配置子网 BGP/EVPN、SDN、交换机 NOS 与自动化工具组合
RDMA 原生 InfiniBand 传输 RoCEv2 或新型 Ethernet 传输
流控 链路级 credit ECN/CNP/端点限速，常配合 PFC
调优重点 路由、VL/QoS、拥塞控制、拓扑 ECN/PFC 阈值、队列、buffer、ECMP、NIC CC
多租户与云融合 更适合独立计算 Fabric 更容易与现有 IP 数据中心融合
厂商选择 相对有限 NIC、交换机、NOS 和光模块选择更多

InfiniBand 的集成度可以降低首次构建专用 HPC/AI Fabric 的不确定性，但运维团队仍然需要理解 SM、LID、P_Key、VL、路由算法和诊断工具。

Ethernet 的开放性可以复用现有 BGP、EVPN、遥测和自动化能力，但“会运维普通 Ethernet”不等于“会运维 RoCE”。RoCE 工程师还需要掌握无损队列、ECN 标记、PFC headroom、CNP、NIC 限速和 GPU 集合通信行为。

八、成本

FiberMall 原文声称 InfiniBand 每端口通常比同速 Ethernet 贵 20%–30%。实际成本会受到以下因素影响：

  • 交换机端口密度与网络层级；
  • 无阻塞或超额订阅比例；
  • NIC/HCA 数量与单机多 rail 配置；
  • DAC、ACC、AOC、光模块和光纤距离；
  • 软件许可、网络管理和遥测系统；
  • 备件、供应周期和厂商折扣；
  • 部署调试与长期运维人力；
  • 网络性能对 GPU 利用率和作业完成时间的影响。

更合理的比较方法是计算总体拥有成本 TCO：

$$
TCO = C_{switch} + C_{NIC} + C_{optics} + C_{cabling} + C_{software} + C_{operation} + C_{downtime}
$$

进一步还应计算每个训练作业或每次仿真的有效成本：如果较便宜的网络导致 GPU 等待时间明显增加，节省的网络资本支出可能被更长的作业时间抵消；反之，如果应用通信强度不高，为极低延迟支付额外成本也没有必要。

九、应该怎样选择？

更倾向 InfiniBand 的场景

  • 构建独立的 HPC 或大规模训练 Fabric；
  • 工作负载高度同步、通信密集，尾延迟直接影响整体进度；
  • 希望使用成熟的一体化 RDMA、GPUDirect 和网内集合通信方案；
  • 更看重可预测性能，而不是与通用 IP 网络融合；
  • 能够接受相对集中的供应商生态。

更倾向 Ethernet/RoCE 的场景

  • 已经拥有成熟的 Leaf-Spine、BGP/EVPN 和自动化运维体系；
  • 需要多厂商选择或与云数据中心网络融合；
  • 同一网络需要承载训练、推理、存储和通用业务；
  • 团队具有 RoCE、PFC、ECN 和 NIC 拥塞控制经验；
  • 工作负载可以接受一定性能波动，或已通过实测证明 Ethernet 满足要求。

十、混合网络往往是更现实的答案

InfiniBand 和 Ethernet 并不一定互相替代。一套 AI/HPC 集群通常同时存在多张网络：

  • InfiniBand 或 AI Ethernet：GPU/CPU 节点间的高性能计算 Fabric；
  • Ethernet Storage Fabric：访问训练数据、检查点和并行存储；
  • Ethernet Management Network：登录、调度、监控、镜像和业务管理；
  • BMC/OOB Network：带外管理、开关机和故障恢复。

图 3：高性能计算网络、存储网络、管理网络和带外网络可以分别选用最合适的技术。

这种分离能够避免普通管理流量干扰集合通信，也允许计算和存储网络采用不同的带宽、拓扑与可靠性策略。是否让存储共享计算 Fabric，需要根据数据路径、故障域、端口预算和作业行为单独评估。

十一、总结

InfiniBand 与 Ethernet 的核心差异，不是“一个快、一个慢”，而是两种不同的系统设计哲学：

  • InfiniBand 从一开始就围绕低延迟、无损运行、硬件 RDMA 和集中式 Fabric 管理构建，适合性能目标明确的专用 HPC/AI 互连；
  • Ethernet 依靠开放标准、广泛生态和灵活的二三层网络成为通用基础设施，并通过 RoCEv2、AI Ethernet 和 Ultra Ethernet 不断吸收高性能计算所需的能力。

InfiniBand 通常提供更一体化、更容易获得可预测性能的路径；Ethernet 通常提供更开放、更易与现有数据中心融合的路径。经过良好设计的 RoCEv2 或新一代 AI Ethernet 可以达到很高的性能，但它并不是“打开 RDMA 开关”就能完成的普通 Ethernet。

最合理的选型原则是：不要让网络技术决定工作负载，而要让真实的通信模式、性能目标、故障模型和运维能力决定网络。
