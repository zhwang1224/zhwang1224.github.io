从 ATOP 到 ZCube：面向大模型训练的数据中心网络拓扑自动优化

论文：From ATOP to ZCube: Automated Topology Optimization Pipeline and A Highly Cost-Effective Network Topology for Large Model Training，SIGCOMM 2025。本文关注一个非常现实的问题：当大模型训练集群扩展到数千、上万张 GPU 时，数据中心网络拓扑应该如何设计，才能在训练性能、硬件成本、故障容忍能力之间取得更优权衡。论文提出了自动化拓扑优化流水线 ATOP，并基于 ATOP 搜索结果发现了一种新的高性价比拓扑 ZCube。论文实验显示，ZCube 相比 ROFT、Rail-only、HPN 等已有拓扑，可以将端到端 LLM 训练速度提升 3%–7%，同时将网络硬件成本降低 26%–46%；真实测试床结果也显示，ZCube 在保持 all-reduce 和 all-to-all 性能相同的情况下，相比 Rail-Optimized Topology 降低约 25% 的硬件成本。

1. 背景

随着生成式 AI 和大语言模型的快速发展，训练集群规模迅速扩大。论文提到，ByteDance 已构建超过 10k GPU 的集群，Meta 为 Llama-3 训练构建了 24k GPU 集群，X.AI 也在 AI 基础设施上投入了巨额资金。随着 GPU 数量增长，网络拓扑不再只是“把 GPU 连起来”的工程问题，而变成一个多目标优化问题。

在大规模训练场景中，拓扑设计至少需要同时考虑三个核心目标：

第一是性能。大模型训练中存在大量 collective communication，例如 all-gather、reduce-scatter、all-to-all，以及 pipeline parallelism 中的点对点通信。如果拓扑直径大、负载均衡差或跨层路径拥塞，就会直接拉长训练 iteration time。

第二是成本。网络成本不仅包括交换机，还包括 NIC、光模块、铜缆、光纤等。对于上万 GPU 集群，网络硬件成本是非常大的投入，因此仅仅追求全带宽或最低延迟并不现实。

第三是故障容忍能力。GPU 集群中单个 ToR switch 或链路故障并不少见。一个拓扑即使在无故障时性能很好，如果一个交换机故障后训练性能大幅下降，也会影响实际可用性。

论文指出，现有常见拓扑如 ROFT、Rail-only、HPN 等在不同维度上各有取舍，但并不一定处于 Pareto 最优前沿。也就是说，可能存在一些拓扑同时在性能和成本上优于它们。ATOP 的目标正是自动化地搜索这些更优拓扑。

2. 大模型训练通信模式

论文首先分析了 LLM 训练中的典型通信模式。大模型训练通常由多种并行策略组成，不同策略产生不同流量：

Data Parallelism，DP 将训练数据切分到不同 GPU，每个 GPU 计算局部梯度，然后通过 all-gather 和 reduce-scatter 等 collective communication 聚合参数或梯度。

Pipeline Parallelism，PP 将模型层切分到不同设备，前向传播时激活值沿 pipeline 传递，反向传播时梯度反向传递，因此产生点对点通信。

Expert Parallelism，EP 常用于 MoE 模型，会产生 all-to-all 流量。

论文基于生产集群中的 4k GPU 训练 trace、SimAI 中的 AICB workload generator，以及对 LLM 研究者和实践者的访谈，发现实际训练中主要有五类流量周期：DP-only、PP-only、EP-only、Mixed DP-PP、Mixed DP-EP。

更重要的是，论文观察到这些流量有一个共同特点：跨服务器通信通常发生在具有相同 GPU index 的 GPU 之间。例如，Server_1 上的 GPU_2 往往与 Server_64 上的 GPU_2 通信。这一特征称为 rail-aligned。即使在 all-to-all 场景中，由于 NCCL PXN 会先通过 NVLink 将流量转发到相同 index 的 GPU，再发往目标服务器，因此也会呈现 rail-aligned 特征。

3. 自动化拓扑设计

传统拓扑设计主要依赖人工专家经验。专家通常会从 HPC 和 DCN 文献中选择已有拓扑，例如 CLOS、Fat-Tree、ROFT、Rail-only、HPN、BCube、Dragonfly 等，然后根据硬件约束进行工程化调整。

但论文认为，人工设计存在三个明显问题：

第一，大规模 GPU 集群的拓扑设计空间极其庞大，人工很难高效探索。

第二，专家往往偏好对称、规整、易解释的拓扑结构，可能忽略一些不直观但性能和成本更优的非对称拓扑。

第三，人工设计很难同时考虑训练性能、成本、故障容忍、硬件约束等多个目标。

朴素的自动化方法也不可行。如果直接用邻接矩阵建模拓扑，对于 N 个节点，搜索空间会呈指数级增长。论文指出，在 16k GPU 规模下，用邻接矩阵搜索拓扑甚至需要数亿个参数，搜索 10 小时仍无法产生连通拓扑。

已有工作 Condor 使用 DSL 和约束求解来辅助拓扑设计，但它仍然依赖专家用约束语言表达设计意图。对于硬件约束，例如每台交换机端口数，DSL 比较容易表达；但对于“最小化 LLM 训练 iteration time”这种应用级端到端性能目标，则很难通过约束语言直观表达。

因此，ATOP 的核心思想是：把专家拓扑设计直觉转化为可搜索的超参数空间，然后用多目标优化算法和高保真仿真器自动搜索拓扑。

ATOP，全称 Automated Topology Optimization Pipeline，由三个主要模块组成：

Topology Hyperparameterization Modeling：将拓扑建模为一组可搜索超参数；
Topology Optimizer：使用优化算法搜索拓扑；
Topology Evaluator：用仿真和理论分析评估拓扑性能。

4. 拓扑建模：从人工直觉到 11 类可搜索超参数

ATOP 的建模方式来自对已有 DCN 和 HPC 拓扑的抽象。论文认为，很多经典拓扑都有两个共性：

第一，存在分层结构。例如第一层是 GPU 或服务器，上层是 ToR、aggregation、spine 等交换机层。

第二，存在层内连接或维度结构。例如 Dragonfly、HyperX、Torus、DCell 等拓扑中，不只是简单上下层连接，还存在同层节点之间的复杂连接模式。

因此，ATOP 的拓扑建模分为两部分：节点分层与层间连接，以及层内连接。

5. 拓扑优化器：NSGA-II 多目标搜索

ATOP 的优化目标可以由用户自定义。论文实验中主要使用 4 类、共 14 个目标：

第一类是 LLM 训练流量下的 job completion time，包括 DP-only、PP-only、EP-only、Mixed DP-PP、Mixed DP-EP 等流量模式。

第二类是 ForestColl all-gather 的理论最优完成时间，用于评估 collective communication 性能。

第三类是单交换机故障下的平均路径长度，用于衡量 fault tolerance。

第四类是网络硬件成本，包括交换机、铜缆、光纤、光模块、NIC 等。

ATOP 使用 NSGA-II 作为多目标优化算法。NSGA-II 是一种经典的多目标进化算法，可以通过 non-dominated sorting 和 crowding distance 在多个目标之间寻找 Pareto-optimal 解集。

在拓扑搜索中，ATOP 并不是寻找唯一最优拓扑，而是寻找一组 Pareto 最优拓扑。用户可以根据预算、性能目标、故障容忍要求，在 Pareto frontier 上选择适合自己的设计。

6. 拓扑评估器：流级仿真 + 两阶段评估

大规模拓扑评估的难点在于仿真成本。NS-3 这类 packet-level simulator 在小规模拓扑上可以使用，但面对数千、上万 GPU 时，时间和内存成本过高。

因此，论文实现了一个 flow-level network simulator。它将 GPU 间通信建模为 flow，并使用 max-min fairness bandwidth allocation 计算带宽分配，同时参考 SimGrid 的方法建模拥塞和交换机延迟。论文在附录中将该 flow-level simulator 与 NS-3 对比，显示在 LLM 训练场景下平均 job completion time 误差约为 1.5%。

为了进一步降低搜索成本，ATOP 使用两阶段评估：

第一阶段，用代表性训练流量片段、故障容忍指标和网络成本快速筛选拓扑，得到 Pareto-optimal topologies。

第二阶段，对第一阶段筛出的 Pareto 最优拓扑运行完整的大模型训练仿真，得到最终端到端训练时间。

论文称，这种两阶段评估可以将 full-scale model training simulation 的数量减少约 95%，显著提升搜索效率。ATOP 在单台 256 核 CPU 服务器上，可以在约 3 天内完成 16k GPU 集群的拓扑搜索。

7. ATOP 的三个主要使用场景

论文重点展示了 ATOP 在三个实际数据中心设计场景中的用途。

7.1 新建数据中心

在新建数据中心场景中，ATOP 分别搜索 256、1024、4096、16384 GPU 规模下的拓扑。搜索结果显示，在不同规模下，ATOP 都能找到训练时间和网络成本之间的 Pareto frontier。

一个重要发现是：在不同 GPU 规模下，位于 Pareto frontier 拐点处的最具性价比拓扑结构非常相似。论文将这类拓扑抽象并命名为 ZCube。

7.2 调整已有数据中心

第二个场景是优化已有 4k GPU 数据中心。初始拓扑是 3-layer non-blocking ROFT。目标是在不购买额外交换机和 NIC 的前提下，通过重新布线提升性能并降低成本。

在这个受限搜索空间中，ATOP 发现 ZCube 同时是性能最优和最具成本效率的拓扑。

7.3 扩容已有数据中心

第三个场景是将已有 1k GPU 数据中心扩展到 4k GPU。ATOP 需要保留原有交换机和 NIC，并考虑新增交换机吞吐限制。同时，论文引入了修改链路数量作为新的优化目标，以降低人工改造成本。

结果显示，ATOP 仍然能够找到训练时间、网络成本和改造成本之间的 Pareto frontier，并且 ZCube 依然是最具成本效率的拓扑之一。

此外，附录还展示了 ATOP 在多租户数据中心和异构 GPU 数据中心中的应用。在 16k GPU 多租户场景下，ZCube 仍位于 Pareto frontier 的拐点，是最具成本效率的拓扑；在 H100/A100 异构 4k GPU 数据中心场景中，ATOP 也能在严格约束下找到优于 ROFT 或更具性价比的拓扑。

8. ZCube：ATOP 发现的新型高性价比拓扑

ZCube 是论文最重要的结果之一。它并不是作者手工设计出的拓扑，而是 ATOP 在自动化搜索过程中发现的一类高性价比网络结构。随后，作者对 ZCube 进行了形式化定义、性能分析与成本评估。

8.1 ZCube 的递归构造

ZCube 采用递归式构造方式。

最小单元 ZCube(n, 1) 是一个由单个交换机连接 n 个 GPU 的结构。

一般地，ZCube(n, k+1) 由以下两部分组成：

  • n 个 ZCube(n, k) 子结构；
  • n^k 个新增交换机。

因此，ZCube(n, k+1) 一共包含：

  • n^(k+1) 个 GPU；
  • k+1 层交换机；
  • 每一层包含 n^k 个交换机。

在 ZCube(n, k+1) 中，每个 GPU 拥有 k+1 个 NIC port，编号为 level-0 到 level-k。GPU 的不同 NIC port 分别连接到不同层级的交换机。

ZCube 的一个重要特点是非对称性。在 ZCube(n, k) 中：

  • level-0 和 level-(k−1) 的交换机需要 2n 个端口；
  • 中间层交换机需要 3n 个端口。

这与 Fat-Tree、BCube、Dragonfly 等传统拓扑通常偏好的对称结构不同。论文认为，这种非对称但高性能的拓扑结构，正体现了自动化拓扑搜索相对于人工设计的价值：它能够发现一些不符合传统设计直觉、但在成本和性能上更优的网络结构。

8.2 ZCube 的直径优势

论文证明，ZCube(n, k) 的网络直径为 k。这意味着 ZCube 可以在较低 hop count 下连接大规模 GPU 集群。

例如，ZCube(128, 2) 使用 256-port 交换机即可连接：

  • 128^2 = 16,384 个 GPU；
  • 网络直径仅为 2。

相比之下，3-layer Fat-Tree 使用 128-port 交换机时可以连接：

  • 524,288 个 GPU；
  • 网络直径为 5。

对于大规模模型训练而言，较低的网络直径有助于减少跨节点通信的 hop count，降低 Pipeline Parallelism（PP）等通信流量的完成时间，从而提升端到端训练性能。

8.3 ZCube 的成本优势

在相同 GPU 规模下，ZCube 相比 ROFT、Rail-only 和 HPN 等拓扑需要更少的交换机，并且可以使用更低带宽的链路。

例如，在 16,384 GPU、51.2 Tbps switch 场景下，论文比较了 ROFT、Rail-only、HPN 和 ZCube 的交换机数量与线缆数量：

拓扑 交换机数量 链路数量与带宽
ROFT 640 台 49,152 条 400G 链路
ZCube 256 台 49,152 条 200G 链路

可以看到，ZCube 不仅显著减少了交换机数量，也降低了单链路带宽需求。

论文总结认为，ZCube 可以带来以下成本收益：

  • 交换机数量减少约 33%–60%；
  • 光模块成本降低约 25%–50%。

这说明 ZCube 在保持较好通信性能的同时，能够显著降低大规模 GPU 集群网络的硬件建设成本。

8.4 ZCube 的故障容忍能力

ZCube 还具备较强的故障容忍能力。

当单个交换机发生故障时，连接到故障交换机的 GPU 可以切换到另一个 NIC port 继续通信。因此，单点交换机故障不会直接导致大量 GPU 完全失联。

论文在 4,096 GPU、GPT-3 175B 训练场景下进行了流级仿真。结果显示，在单个 ToR 交换机故障情况下：

拓扑 训练性能下降
ZCube 2.8%
ROFT 46.9%
HPN 9.0%

可以看到，ZCube 在单交换机故障下的性能损失显著低于 ROFT 和 HPN。

此外，由于 ZCube 所需交换机数量更少，其潜在故障点数量也相对更少。这进一步提升了 ZCube 在大规模训练集群中的可靠性和部署吸引力。

总结

我认为这篇论文的价值可以概括为四点。

第一，它没有只提出一个新拓扑，而是提出了一个自动化拓扑优化框架 ATOP。这使得论文不只是针对某个固定集群配置做手工设计，而是提供了一种面向不同规模、不同硬件约束、不同 workload 的通用搜索方法。

第二，它将已有 DCN 和 HPC 拓扑中的专家经验转化为可搜索超参数空间。这一步非常关键，因为它避免了邻接矩阵搜索空间过大的问题，同时又比纯人工设计更灵活。

第三，它提出了一个工程上可行的两阶段评估机制。通过先用代表性 workload fragment 筛选 Pareto-optimal topologies，再进行完整训练仿真，ATOP 在可接受时间内完成了 16k GPU 规模拓扑搜索。

第四，它发现并形式化了 ZCube。ZCube 的意义在于，它说明大规模 GPU 网络拓扑不一定必须沿着传统对称结构设计。非对称拓扑如果能降低直径、减少交换机和光模块成本，并保持 collective communication 性能，就可能在真实大模型训练集群中更具性价比。
