Alibaba HPN: A Data Center Network for Large Language Model Training

HPN（High Performance Network），即阿里云专为大语言模型（LLM）训练设计的数据中心网络。由于 LLM 与通用云计算在流量模式和容错性等方面存在显著差异，传统数据中心网络难以有效支撑 LLM 训练需求。这促使我们专门针对 LLM 训练设计一种全新的数据中心网络架构。

与通用云计算产生数百万个小流量（例如低于 10 Gbps）不同，LLM 训练在每台主机上仅产生少量周期性、突发性的大流量（例如 400 Gbps）。这种特性使得传统数据中心广泛采用的等价多路径（ECMP）负载均衡机制容易出现哈希极化（hash polarization），导致流量分布不均等问题。

HPN 提出了一种两层双平面架构，可在单个 Pod 内互联多达 15,000 块 GPU——这一规模通常需要传统三层 Clos 架构才能支持。该新架构不仅通过减少 ECMP 的使用避免了哈希极化问题，还大幅缩小了路径选择的搜索空间，从而能够精确挑选适合承载“大象流”（elephant flows）的网络路径。

LLM 训练面临的另一挑战在于：其要求所有 GPU 同步完成迭代，因此对单点故障（尤其是机架顶部交换机 ToR 故障）更加敏感。为此，HPN 提出了一种新型双 ToR 设计，替代传统数据中心中的单 ToR 方案，并解决了二层同步方面的技术难题。HPN 已在我们的生产环境中部署运行超过八个月。

1. Motivation

LLM 训练具有独特性，它对数据中心网络的设计提出了全新挑战。

  • 问题一：流量模式不同。
    LLM 训练的流量模式与通用云计算存在两点关键差异：(1) 低熵性；(2) 突发性流量。具体而言，通用云计算会产生数百万个持续且低利用率的流量（例如通常低于网卡容量的 20%），如图 1 所示。
  • 问题二：对故障（尤其是单点故障）更为敏感。
    LLM 训练是一个同步过程，所有 GPU 协同完成一系列迭代；因此，任意一块 GPU 出现异常都可能导致整个训练过程延迟甚至崩溃。这意味着 LLM 训练比通用云计算对故障更加敏感。我们发现，对 LLM 训练影响最大的是与机架顶部交换机（Top-of-Rack, ToR）相关的单点故障，这类故障可能波及大量 GPU。此外，LLM 训练中的故障代价高昂。生产数据显示，LLM 训练中的一次故障造成的损失是通用云计算场景下的 20 倍。

主要目标：

  • G1：可扩展性。 图 6 显示，当前生产中单个训练任务所需 GPU 数量通常少于 3,000 块。为应对未来需求演进，我们将主容量目标设定为支持 15,000 块 GPU——这也与主流 LLM 训练提供商（如 Google、AWS、Azure 和 NVIDIA）提供的 10K–30K GPU 集群规模一致。根据经验，未来几年模型参数量可能再增长一个数量级（从 1 万亿增至 10 万亿），因此我们的数据中心还需具备支持 10 万块 GPU 规模的扩展能力。
  • G2：高性能。 性能至关重要。为提升性能，设计应尽可能减少网络跳数。减少跳数不仅能降低延迟，还能减少 ECMP 哈希次数，使路径选择更精准。此外，设计应尽可能支持 GPU 间直连通信，而非经由网络转发。
  • G3：单 ToR 故障容错。 基于 §2.3 的观察，影响 LLM 训练可靠性的最关键风险是单 ToR 故障。因此，我们的新网络架构必须在拓扑层面从根本上消除单 ToR 故障风险。

2. HPN 架构概述

图 7 展示了 HPN 的整体架构。HPN 由前端网络（frontend network）和后端网络（backend network）组成。后端网络主要用于承载训练过程中的通信流量；前端网络则负责其他类型的流量（例如管理、推理和存储流量）。在 LLM 训练场景中，我们主要关注 HPN 的后端网络。

  • 在 HPN 的后端网络中，每台主机配备 8 块 GPU，这些 GPU 通过专用的高带宽主机内网络（如 NVLink ）相互连接。每块 GPU 可通过该主机内网络（intra-host network）以 400–900 GBps（双向）的带宽直接与其他 GPU 通信。
  • 为了提供最大网络容量，每台主机配备了 9 张网卡（NIC），每张 NIC 具备 2×200 Gbps 的端口。其中一张 NIC（即图 7 中的 NIC0）连接到前端网络，其余 8 张 NIC 连接到后端网络，用于承载 LLM 训练期间的通信流量。
  • 这 8 张 NIC 各自服务于一块专用 GPU（称为一个“rail”），因此每块 GPU 都拥有 专用的 400 Gbps RDMA 网络吞吐能力，整机总带宽达到 3.2 Tbps。这种设计旨在最大化利用 GPU 的 PCIe 能力（PCIe Gen5×16），从而将网络发送/接收能力推至极限。
  • 每张 NIC 的两个端口分别连接到不同的 ToR 交换机，形成 双 ToR（dual-ToR）。该设计旨在避免单 ToR 故障问题。

在 Tier-1 层，考虑到主机内网络（如 NVLink）的带宽远高于主机间的以太网，我们采用了 rail 优化设计：不同 rail 的 NIC 连接到不同的 ToR 集合。结合上述双 ToR 设计，一台主机在后端网络中总共连接到 16 台 ToR 交换机。通过充分利用 51.2 Tbps 交换芯片的能力，HPN 能够在一个单层网络（称为一个 segment，见 §5）中互联 1024 块 GPU。图 6 量化了这一优势：约 96.3% 的线上 LLM 训练任务使用的 GPU 数量少于 1000 块；因此，在 HPN 中，这些任务可以被部署在同一个 segment 内，从而获得极致的网络性能。

Tier-2 用于互联多个 segments。我们在这一层引入了双 ToR 特性，设计了 双平面转发架构（dual-plane forwarding，如图 7 所示）。从源 NIC 的 port 0 发出的流量，只能通过网络最终被目的 NIC 的 port 0 接收，与从 port 1 发出的流量在物理上完全隔离。这种双平面设计避免了聚合层（Aggregation layer）的 哈希极化（hash polarization）问题，同时保持了 1:1 的网络对分带宽（bisection bandwidth）。此外，双平面设计还将单个 Pod 可支持的 GPU 数量提升了一倍，达到 15,000 块 GPU。

对于未来可能出现的更大规模单任务，我们还设计了 Pod 之间的核心层（Core layer）。由于单个 Pod 已支持 15K GPU，需要跨 Pod 协同的任务非常罕见。在我们的设计中，聚合层到核心层之间采用了 15:1 的带宽超配比（oversubscription）。基于 LLM 训练的流量特征，我们将 流水线并行（PP）通信安排在跨 Pod 之间，确保跨 Pod 传输对端到端训练性能的影响最小化。

前端网络主要用于承载管理、推理和存储流量。前端与后端网络在物理上解耦，确保前端流量不会干扰训练任务的核心流程。此外，前端网络采用 1:1 无超配设计，使其具备良好的可扩展性，能够支持更多场景，例如 LLM 训练与推理的混合部署。

3. Non-stacked Dual-ToR

在传统数据中心网络中，每张 NIC 的两个端口通常通过一根线缆连接到同一台 ToR 交换机，称为 单 ToR 设计。然而，单 ToR 设计对交换机或链路故障极为敏感，会严重影响 LLM 训练。相比之下，双 ToR 设计将每张 NIC 的两个端口以 active-active 方式连接到两台不同的 ToR。这两个端口配置相同的 IP 和 MAC 地址。当其中一台 ToR（或端口）发生故障时，另一台仍可正常工作。此外，由于同一张 NIC 的两个端口共享相同的 Queue Pair（QP）上下文，流量切换不会中断现有数据流，对上层应用完全透明。

图 8a 展示了一种典型的双 ToR 设计，称为 堆叠式双 ToR（stacked dual-ToR）。但ToR 之间的直连链路故障或某台 ToR 故障可能导致异常，使该双 ToR 下所有 NIC 离线。在双 ToR 升级过程中，可能出现一台运行新版本、另一台运行旧版本的情况。此时，控制平面通过 RPC 同步信息时，因版本字段不兼容，可能导致 ToR 崩溃。根据内部故障报告，过去三年中，超过 40% 的关键故障源于上述两类堆叠式双 ToR 引入的问题。

为消除这一故障根源，我们设计了 非堆叠式双 ToR（图 8b），移除了两台 ToR 之间的直连链路。为实现非堆叠双 ToR，我们必须保证在 LACP 协商过程中，两台 ToR 使用相同的 MAC 地址和不同的 portID。

4. Tier1

使用了最新的 51.2Tbps 以太网单芯片交换机（最早于 2023 年初发布）在 HPN 中。在 tier1 层，每台交换机拥有 128 个活跃 + 8 个备用 200Gbps 下行端口和 60 个上行 400Gbps 端口。这种设计确保了一个接近 1:1 的超配比（实际为 1.067:1）。每个 ToR 交换机预留了 8 个备用下行端口，用于连接备用主机，以便在主机侧故障时（包括 CPU、内存、GPU、PCIe、NVLINK 和 NIC 故障）快速替换。

为了充分利用不同的转发能力，NVIDIA 首次提出了 rail 优化网络的概念，该概念已在训练集群中广泛应用。在 rail 优化网络中，相同 rail 的 NIC 通过同一组双 ToR 交换机连接。不同 rail 的 NIC 可以通过主机内+主机间转发组合进行通信。例如，在图 11 中，如果 host1 中的 GPU1 想要与 host3 中的 GPU2 通信，转发路径将是：host1 中的 GPU1 → host1 中的 GPU2 → ToR3 → host3 中的 GPU2。

图 11 展示了我们在实践中如何应用 rail 优化网络，这使得单个主机的 3.2Tbps（8×400Gbps） 可以分布在多达 16 个 ToR 交换机 上（在双 ToR 设计下）。与原始拓扑相比，一个 segment 可容纳的 GPU 数量增加了 8 倍，在原始拓扑中，单个主机的 3.2Tbps 是由 2 个 ToR 交换机提供的。每组双 ToR 交换机可以服务 128 块 GPU，16 个 ToR 共同连接了 segment 中的 1024 块 GPU，大幅减少了转发延迟并提供了最佳性能。更重要的是，它显著减少了穿越聚合层的流量，降低了网络负载不平衡的可能性。

5. Tier2

在 tier1 网络中采用双 ToR（dual-ToR）设计后，如果 tier2 层简单地使用传统的 Clos 拓扑（如图 12a 所示），仍然会出现 哈希极化（hash polarization） 问题。在 ECMP（等价多路径）路由中，流量根据五元组（5-tuple）哈希分配到不同路径。但由于哈希算法的局限性，大量流可能被映射到少数几条路径上，导致某些链路过载、其他链路空闲——这就是“哈希极化”。

在下游方向（Aggregation → ToR），由于每个 dual-ToR 组只有 2 台 ToR，而上游有 60 台 Aggregation 交换机汇聚流量，极易造成流量集中。如图 13a 所示，在真实 GPT-3 175B 变体模型训练任务中，通往同一 NIC 的两个 ToR 下行端口流量相差 3 倍，严重拖累训练性能。

如图 12b 所示，HPN 将每个 dual-ToR 组中的 ToR 划分为 两个独立平面（Plane 1 / Plane 2）：每个流一旦从某 ToR 的上行链路进入 Pod，其在整个 Pod 内的转发路径就被唯一确定。两个平面物理隔离，互不干扰。

6. 支持更大规模

为满足未来支持 十万个级别 GPU 的长期规划，HPN 设计了 tier3 网络，用于连接多个 Pod。

⚖️ 权衡：Pod 规模 vs. tier3 带宽
当前 HPN 牺牲了 Agg-Core 层的带宽（超配比 15:1），以换取更大的单 Pod 规模；但分析 LLM 训练通信模式发现：跨数十万 GPU 的单个训练任务，对 tier3 带宽需求其实很低。

为避免 tier3 引入新的不均衡风险，HPN 做了两项增强：1. Core 层延续 dual-plane 设计；2. 每台 Core 交换机采用“端口优先哈希”。

7. 前端网络

在 HPN 中，每台主机额外配备一块 2×200Gbps 的 NIC，用于接入独立的前端网络。前端网络主要承载管理流量和存储流量。为保障最优训练性能，存储集群被部署在前端网络中。

HPN 的核心优势：

  • 消除 single-ToR 单点故障；
  • 通过 两层 dual-plane 架构，在单 Pod 内互联 15,000 块 GPU；
  • 彻底解决哈希极化问题；
  • 简化最优路径选择；
  • 端到端 LLM 训练性能提升 14.9%。
