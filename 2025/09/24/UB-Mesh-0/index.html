UB-mesh

1. LLM 训练对数据中心网络的四重压力

随着大语言模型（Large-scale Language Models，LLM）沿着缩放定律（Scaling Laws）持续扩张参数规模与训练数据量，底层训练系统所承受的压力被同步放大。论文将下一代 AI 数据中心需要同时满足的诉求归纳为四点：

  1. 大规模 (R1) —— 需支持数十万级 NPU/GPU。
  2. 高带宽 (R2) —— 单节点需要超过 3.2 Tbps 互连，总带宽比传统 CPU 系统高 10–100 倍。
  3. 高性价比 (R3) —— 传统对称 Clos 架构过于昂贵，需要新的网络设计以降低 CapEx（硬件资本开销） 与 OpEx（运维开销）。
  4. 高可用性 (R4) —— 在 10 万 GPU 集群下原始 MTBF 也将缩短至 不到 30 分钟，系统必须具备容错与自愈能力。

为应对上述挑战，作者提出了 三大设计原则：

  • P1: 拓扑需贴合 LLM 流量模式，充分利用通信局部性。例如，Tensor Parallelism 产生的集体通信占总流量 50% 以上，且集中于 8–64 个相邻 NPU；而数据并行带来的长距离通信不足 2%。因此，需要分层、局部化的网络架构。
  • P2: 在分层局部化的网络上，并行策略、路由、集合通信与负载均衡都必须与网络拓扑精细对齐，否则会因拥塞或带宽利用不足而性能低下；
  • P3: 在链路或 NPU 失效时，路由系统应自动切换至备用路径、备用 NPU 应能无缝接管，从而保障训练的连续性。

为满足 R1–R4 并遵循 P1–P3，作者提出 UB-Mesh 架构：

  • 采用 分层局部化的 nD-FullMesh 拓扑，优先短程直连，减少交换机与光模块依赖；
  • 实现 分层带宽分配，满足不同范围的通信需求；
  • 引入 全路径路由 (APR) 与拓扑感知容错机制，结合 64+1 高可用设计，提升性能与可靠性。

实验结果表明：

  • UB-Mesh 相比非阻塞 Clos 架构，节省 98% 高端口交换机与 93% 光模块；
  • 在性能损失不足 7% 的情况下，系统级成本效率提升 2.04 倍，可用性提升 7.2%；
  • 在多种 LLM 训练任务中保持 95% 以上的线性扩展性。

2. 核心洞察：LLM 训练流量的分层局部性

2.1 通信墙问题 (Communication Wall)

LLM 训练规模空前，需要成千上万的 NPU/GPU。每次训练迭代包含前向、反向和参数更新，大量依赖 跨 NPU 的数据交换。随着训练系统的规模增大，数据通信开销成为系统中最昂贵的部分。随着集群规模扩大，通信成本成为主要瓶颈，即所谓的“通信墙”。

2.2 数据流量的局部性 (Locality of Data Traffic)

其次，LLM 训练通常并行使用多种并行化技术，而这些技术产生的数据流量并不均等。基于一个内部 MoE-2T 模型的分析，论文给出了如下流量分布。

TP 与 SP 合计约占总流量的 97%，而其余并行技术通常产生不足 2% 的流量。更重要的是，TP 与 SP 引发的集合通信往往发生在 8 至 64 个相邻 NPU 构成的簇内，呈现短程、密集的特征；而 DP 引发的通信虽占比极低（不足 2%），却通常需要长程传输。换言之，LLM 训练流量在物理空间上是分层且高度局部化的。因此，架构设计应当优先采用层次化方式来提供网络带宽。

2.3 现有数据中心架构的局限性

  • Clos：性能高，但对称互连成本过高（大量高性能交换机与光模块）。
  • 3D Torus：依赖直连，成本较低，但带宽不足，难支持复杂 All-to-All 通信。
  • DragonFly：比 Clos 便宜，但仍需全带宽交换，短程通信也需经过交换结构，对 LLM 特征流量表现不佳。
  • Fugaku Tofu (6D Torus)：适合 HPC，但与 3D Torus 类似，难以承载 LLM 的通信模式。

3. UB-Mesh 架构 (Architecture)

3.1 nD-FullMesh 拓扑

  • 核心思想：优先使用电缆直连在各层次构建全互连域，逐层递归构造 full-mesh，形成 分层局部化 的拓扑。
  • 构建方式：
    • 1D-FullMesh：板内 NPU 全互连；
    • 2D-FullMesh：机架内相邻 1D 域互连；
    • 3D-FullMesh：跨机架互连；
    • 4D/更高维：跨机房/楼层/数据中心互连。
  • 优势：
    • 充分利用数据局部性与流量模式：nD-FullMesh 在每个网络维度上形成紧耦合的直连域，能够逐层提供超配带宽，从而充分利用 LLM 训练中固有的数据局部性和由密到疏的流量分布特性。还可以灵活调整每个维度的节点带宽分配；
    • 利用数据局部性：直连拓扑显著缩短了传输距离：结合数据放置与集合通信优化，大多数传输（典型即 TP 或 SP）可在 0 至 2 跳内完成，从而大幅压低数据移动开销；
    • 减少交换机/光模块依赖：相比 Clos，nD-FullMesh 削弱了对高基数交换机与长程光互连的依赖，转而最大化短程直连；
    • 支持复杂集体通信：短程无源电缆占据了全部线缆消耗的 86.7%。由于电缆与连接器比光模块更稳定，这一设计在削减交换机与光模块成本的同时还提升了系统可靠性。此外，与同样依赖直连的 3D-Torus 不同，全互连拓扑能够高效支持 all-to-all 这类复杂的集合通信操作，弥补了 Torus 类拓扑在 MoE 场景下的短板。。

3.2 UB-Mesh 的基本构件

  • 硬件模块：
    • NPU：AI 计算核心，配备 2×UB 控制器 (72 lanes)；
    • CPU：主机处理，配备 1×UB 控制器 (32 lanes)；
    • LRS (低端口数交换机)：轻量化，主要做机架内/相邻聚合；
    • HRS (高端口数交换机)：大规模互连，Pod 级聚合。

  • Unified Bus (UB) 技术：
    • 单一互连机制，替代 PCIe / NVLINK / IB / RoCE 的混合方案；
    • 优势：
      1. I/O 灵活分配（近端粗带宽 / 远端稀疏带宽）；
      2. 硬件资源池化（CPU、NPU、DDR、NICs 可灵活绑定）；
      3. 系统优化简化（协议统一，减少驱动/库的转换开销）。

3.3 架构实现 (UB-Mesh-Pod)

通过将表 3 中列出的硬件构件组合在一起，并使用统一总线（UB）进行互连，文章提出了 UB-Mesh 的整体架构设计。如图 7-(a) 所示，文章按照所提出的 nD-FullMesh 拓扑实现了一个 UB-Mesh-Pod。具体来说，在该架构中，在每个机架内部构建一个 2D-FullMesh，并将其扩展为跨机架的另一个 2D-FullMesh，最终形成 4D-FullMesh。在本代系统中，选择不将 UB-Mesh 扩展到 5D 或更高维度，以在成本效率与灵活性之间取得工程上的平衡；在 4D-FullMesh 之外，仍采用 Clos 拓扑。

  • 机架级 (Rack)：
    机柜内的 2D-FullMesh 是构建的起点。每个机柜由 8 块 NPU 板卡组成，每板卡含 8 个 NPU，合计 64 个 NPU，经由电缆直连构成二维全互连（X-Fullmesh 与 Y-Fullmesh），以保障 NPU 间的高带宽。与 CPU、NPU 同板的传统设计不同，UB-Mesh 将 CPU 板卡独立出来，通过交换机与 NPU 相连，从而支持灵活的 CPU/NPU 配比以及 CPU/NPU/DDR 的资源池化。机柜内多块背板交换机（即 LRS）以非阻塞方式管理柜内与柜间连接，聚合输出四路 UB x256 IO。

  • 64+1 高可用设计：
    64+1 高可用设计 是机柜层面的容错机制。系统在 64 个常规 NPU 之外额外配置一个经由 LRS 接入的备用 NPU。当某个 NPU（如图示中的 NPU-3）失效时，管理系统激活备用 NPU 接管其角色，与该 NPU 相关的直连链路被重定向——例如原本的「5→3」路径被改写为「5→LRS→B」。这一策略将原先的一跳直连改为经交换机的一跳路由，略微增加了传输时延，但相比简单屏蔽故障 NPU、仅以剩余七个 NPU 运行任务的方案要优越得多；在以柜内带宽为首要考量的 LLM 训练场景下，这点额外时延对整体训练性能影响可忽略。

  • Pod 级 (UB-Mesh-Pod)：
    Pod 级的 4D-FullMesh 在机柜之上再叠加一层二维全互连。一行内的四个相邻机柜经各柜 LRS 端口直连构成一维全互连，机柜列方向同样直连，最终形成 16 机柜的二维全互连；此处选择沿两个维度连接四个相邻机柜，是综合考虑有源电缆可达距离后的最优点，每条链路提供 UB x128 IO。由于每柜 64 NPU、每 Pod 16 柜，一个 4D-FullMesh 的 UB-Mesh-Pod 共含 1024 个 NPU。论文给出的带宽分配是：柜内 2D-FullMesh 约 UB x56/NPU，柜间 4D-FullMesh 约 UB x12/NPU。

  • SuperPod / DCN 扩展：
    SuperPod 与更大规模 则在 Pod 之上改用对称拓扑。考虑到云场景中中小规模训练任务未必占满整个 SuperPod，论文在 Pod 级互连上不再延续全互连，而是采用对称的 Clos 拓扑，以高基数 Pod 交换机（HRS）连接各机柜，使云管理者能按需切分 SuperPod 并在每个划分域内保证满带宽，规模可达 8K NPU（约 UB x4/NPU）。再向上，SuperPod 内的机柜通过 UB 交换机（方案 a：UB-to-DCN）或 CPU 板卡上的 NIC（方案 b：NIC-to-DCN）接入大规模数据中心网络（Data-Center Network，DCN），DCN 通常承载大规模数据并行训练，以 Clos 组织，可扩展至 10 万 NPU 乃至更多。

由此，UB-Mesh 形成了「4D-FullMesh + Clos」的混合拓扑：短程、高占比流量走直连全互连，长程、低占比流量走交换式 Clos，与第二节揭示的流量结构精确对应。

4. 架构实现 Enabling The Architecture

「4D-FullMesh + Clos」的混合拓扑对路由系统提出了五项要求：支持混合拓扑、高效转发（因每个 NPU 既是计算单元又是路由器，须节省其硬件资源）、支持非最短路径（以充分利用全互连提供的多条路径来最大化带宽利用）、快速故障恢复、以及无死锁。论文指出，Clos、Torus 与 DragonFly 上既有的最长前缀匹配（LPM）、基于主机的路由、维序路由（Dimension Ordered Routing，DOR）等技术均无法同时满足这五项要求，故提出 全路径路由（All-Path Routing，APR） 与基于直接通知的快速故障恢复。

4.1 全路径路由（All-Path Routing, APR）

UB-Mesh 的全互连网络架构在任意两个 NPU 之间提供多条路径。传统的路由策略（如最短路径优先 SPF，见图 10-(a)）往往导致网络带宽利用不足，并且容易受到链路故障的影响。为提升网络效率与鲁棒性，文章提出了 全路径路由（APR） 机制。

APR 的出发点是：全互连架构在任意两个 NPU 间提供了多条路径，而传统的最短路径优先（Shortest-Path First，SPF）路由既不能充分利用带宽，又易受链路失效影响。APR 转而利用源与目的之间的全部可用路径，实现动态路径切换以应对故障或拥塞。它依托三项底层机制。

  • 源路由（Source Routing, SR）机制：实现自适应路由。
    第一是源路由（Source Routing，SR）。在原始包头之外附加一个 8 字节的 SR 头，其中包含转发指令。每个路由器用一个 4 比特的指针（ptr）在 12 比特的位图（bitmap）字段中定位偏移，第 ii 个比特决定第 ii 跳的转发方式（置 1 表示按 SR 转发，置 0 表示传统转发）；当按 SR 转发时，位图还用于定位六个指令字段之一，指示具体如何转发。SR 信息高度紧凑，开销很低。

  • 结构化寻址与线性表查找：降低路由表查找和转发开销。
    为压低每个 NPU 内 UB IO 控制器的路由表查找与转发开销，APR 依据网络元素（Pod、机柜、板卡）的物理位置将寻址空间划分为段。同一段内的 NPU 共享相同前缀，因此只需存储短段地址，段内 NPU 通过相对于段地址的线性偏移寻址。这一设计显著减小了表空间，加快了路由表的生成、分发与收敛，并使故障恢复等状态变更能够被迅速响应。

  • 拓扑感知的无死锁流控（TFC）：保证全路径路由下的死锁规避。
    由于 nD-FullMesh 含有复杂的环结构、且 APR 允许多跳路由，在有限虚拟通道（Virtual Lane，VL）资源下实现无死锁流控颇具挑战。TFC 用通道依赖图（Channel Dependency Graph，CDG）对死锁建模，将拓扑划分为子图，在各子图内统一拓扑导向规则与 VL 约束，并施加 N 维跨维度断环原则将其分解为单 VL 无环子集，再以同维断环原则计算幂集元素的排列与笛卡尔积，由此生成全路径组合与 VL 映射。其结果是：仅用 2 个 VL 资源即可实现无死锁的全路径路由。

4.2 性能最大化：拓扑感知的集合通信与并行化

当网络链路发生故障时，传统路由系统通常采用逐跳通知机制。如图 12 左侧所示，当链路 1-3 出现故障时，该信息由节点 1 和 3 逐跳传播，通常会消耗较长延迟。

在 UB-Mesh 中，由于每个节点都有确定的一组通信目标，文章可以通过直接通知这些节点来加速路由收敛。如图 12 右侧所示，当链路 1-3 出现故障时，该信息会根据预先计算的路由关系直接发送到节点 6。凭借这种 拓扑感知的直接通知，控制平面的开销大幅减少，从而显著加快了故障恢复。

5. 最大化性能（Maximizing The Performance）

本章节主要针对训练任务的分布以及集合通信的性能进行优化。文章提出了若干拓扑感知的优化策略，以提升性能表现。

5.1 拓扑感知的集合通信（Topology-Aware Collective Communication）

All-Reduce 与 All-to-All 两个典型操作为例。

拓扑感知的集合通信 充分利用 APR 提供的多路径能力。对于 AllReduce，论文提出多环（Multi-Ring）算法：先以统一抽象对网络拓扑建模（计入节点数、节点间连接、链路带宽与时延），再用逻辑多环算法将集合通信与路径映射结合、确保各环路径互不冲突，并将默认映射之外的空闲链路经 APR 借用以增益带宽，最后在多条路径间做最优流量划分以缓解瓶颈。对于 All-to-All，论文区分两种典型场景：在一般 All-to-All 中采用多路径 All2All，将每个元素切分为两部分，分别沿 X-FullMesh 与 Y-FullMesh 同时传输并最多一跳到达，从而保证高带宽利用；在涉及 token 分发与专家数据收集的 All-to-All 中，其语义等价于多个 broadcast 与 reduce 的叠加，于是采用分层广播/归约以节省带宽。

5.2 拓扑感知的并行化（Topology-Aware Parallelization）

提出 拓扑感知并行化机制，用于优化 LLM 训练中的模型与数据划分。其流程如图 15 所示：

  • 步骤 1：生成可行的并行化配置，并将其映射到 UB-Mesh 架构；

  • 步骤 2：利用 拓扑感知通信开销模型 对通信成本进行评估；

  • 步骤 3：通过迭代最小化通信开销，找到最优配置。

构建合适的搜索空间，以平衡效率与性能。

  • 采用基于优先级的启发式剪枝：TP 与 SP（或 CP） → 通信量大，优先映射到高带宽域；PP 与 DP → 用于梯度更新，优先级最低；对于 MoE 模型需要的 EP，我们强制 SP×DP 为 EP 的整数倍。
  • 确保开销模型足够准确: 精确建模 APR 与拓扑感知集体通信在 UB-Mesh 拓扑下的行为；并利用高精度的内部仿真平台来校准该模型。

6. 性能评估

文章分为机架内和机架间两个层次，进行比较。探索 UB-Mesh 的架构设计空间，并分析其相较基线 Clos 架构的优势。

6.1 实验设置

  • 基准负载：基准工作负载包括稠密模型和稀疏模型。如LLAMA-70B 与 GPT3-175B 是稠密模型，而 GPT4-2T 是采用 MoE 技术的稀疏模型。
  • 仿真平台：自研的仿真平台，该平台与实际 PoC 硬件保持一致，用于评估集群规模下的 LLM 训练性能。

6.2 机架内架构探索（Intra-Rack Architecture Exploration）

架构对比（见 Fig. 16）：

  • (a) 2D-FM：

    • UB-Mesh 的设计，64 NPUs 电缆直连，形成 2D 全互连拓扑 (X-FM + Y-FM)。
    • 成本低，使用 48 LRS 聚合跨机架带宽与 CPU-NPU 通信。

  • (b) 1D-FM-A：

    • 保留板内 1D X-FullMesh（8 NPU 直连）；
    • 跨板通信由 36 LRS 完成；
    • 每个 NPU：UB x16 → LRS，另 UB x16 → HRS（跨机架通信）。

  • (c) 1D-FM-B：

    • 用 HRS 替代部分 LRS；
    • 背板中 4 LRS 专用于 NPU-CPU 通信；
    • 跨板 NPU 通信由 8 个交换机完成；
    • 每个 NPU 获得 UB x32 的跨机架带宽。

  • (d) Clos：

    • 所有 64 NPUs 端口接入 72 LRS，形成对称 Clos；
    • 提供最高灵活性，但交换机资源消耗极大。

性能比较（见 Fig. 17）：

  • 2D-FM vs Clos：性能为 93.2%–95.9%；
  • 1D-FM-A：比 2D-FM 在 LLAMA2-70B 上提升 2.44%；其他大模型提升 <1.6%；
  • 1D-FM-B：比 2D-FM 提升 >3%，但整体提升仍有限；
  • 序列长度分析：
    • 8K–32K：2D-FM = 95.5%，略低于 1D-FM-A (98.1%)、1D-FM-B (99.2%)；
    • 64K–10M：2D-FM = 95.0%。

✅ 结论：与 Clos 相比，2D-FM 在显著降低硬件成本的情况下，性能差距控制在 7% 以内。

6.3 机架间架构探索（Inter-Rack Architecture Exploration）

架构对比（见 Fig. 18）：

  • (a) 2D-FM：

    • 16 个机架在水平/垂直方向直接互连，形成 2D-FM；
    • 所有机架也连接到 HRS，用于跨 Pod 互连；
    • 支持三种路由策略：
      • Shortest：仅最短路径；
      • Detour：利用 APR，使用更多路径；
      • Borrow：通过交换机“借用”带宽。

  • (b) Clos：

    • 所有机架接入 HRS，无直连；
    • 消耗交换机更多，但提供最高带宽与灵活性。

性能比较（见 Fig. 19）：

  • GPT3-175B：对跨机架通信不敏感；
  • GPT4-2T：
    • Shortest 策略下，2D-FM 比 Clos 低 0.73%；
    • Detour+Borrow 策略下，差距缩小到 0.46%；
  • 总体：2D-FM ≈ Clos，性能几乎相当。

跨机架带宽探索（见 Fig. 20-(b)）：

  • 评估带宽：UB x4 / x8 / x16 / x32 per NPU；
  • 8K–32K 序列：最佳带宽 = UB x16；
    • 从 UB x8 → x16，仅提升 0.44%；
  • 64K–10M 序列：最佳带宽 = UB x32；
    • 从 UB x16 → x32，性能提升 1.85%；

➡️ 长序列下 TP/SP 流量不可避免跨机架，高带宽显著降低通信延迟 → 提升性能。

默认设置：

  • UB-Mesh 默认为每个 NPU 配置 UB x16 I/O 用于跨机架通信；
  • 可调整机架内/机架间带宽比例，以适配不同 LLM 训练任务需求。

7. 创新与贡献总结

  • 在拓扑层面，论文提出 nD-FullMesh 这一递归构建、分层局部化的全互连拓扑，并将其落地为「4D-FullMesh + Clos」的 UB-Mesh-Pod。其根本贡献在于，把 LLM 训练流量「短程密集、长程稀疏」的结构性特征直接编码进网络拓扑，使高占比的 TP/SP 流量在 0 至 2 跳内的直连电缆上完成，从而以无源电缆占比 86.7% 的物理介质结构替代了 Clos 对高基数交换机与长程光模块的重度依赖。

  • 在硬件与互连层面，论文以统一总线 UB 取代 PCIe/NVLink/IB/RoCE 的异构混合互连，配合 NPU、CPU、LRS、HRS 四类器件，实现了灵活的 IO 资源分配、跨组件的硬件资源池化与跨层优化，并让每个 NPU 兼任路由器。

  • 在网络系统层面，论文提出全路径路由 APR，以源路由、结构化寻址与线性表查找、拓扑感知无死锁流控三项机制，满足了混合拓扑、高效转发、非最短路径、快速恢复与无死锁的全部要求，其中 TFC 仅以 2 个 VL 实现无死锁全路径路由、直接通知机制压缩了故障恢复时延，均为面向该拓扑的针对性设计。

  • 在系统与算法层面，论文给出拓扑感知的集合通信（多环 AllReduce、多路径与分层 all-to-all）与拓扑感知的并行化（基于优先级的搜索 + 经校准的通信代价模型），将硬件能力转化为可观的端到端训练性能。

  • 在工程效果层面，相较非超额订阅的 Clos，UB-Mesh 减少 98% 的高基数交换机与 93% 的光模块，取得 2.04× 的系统级成本效率提升与 7.2% 的可用性提升，性能损失控制在 7% 以内，并在多个 LLM 训练任务上达成 95% 以上的线性度。这表明，面向 LLM 训练这一具有确定性、强局部性流量的特定负载，以流量模式为先导、放弃对称带宽假设的专用网络架构，能够在成本、可用性与性能之间取得优于通用 Clos 的整体平衡。
