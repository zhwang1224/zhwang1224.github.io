UB-mesh

1. Introduction

论文指出，大规模语言模型（LLMs）的训练对数据中心提出了 四大关键需求：

  1. 大规模 (R1) —— 需支持数十万级 NPU/GPU。
  2. 高带宽 (R2) —— 单节点需要超过 3.2 Tbps 互连，总带宽比传统 CPU 系统高 10–100 倍。
  3. 高性价比 (R3) —— 传统对称 Clos 架构过于昂贵，需要新的网络设计以降低 CapEx（硬件资本开销） 与 OpEx（运维开销）。
  4. 高可用性 (R4) —— 在 10 万 GPU 集群下原始 MTBF 也将缩短至 不到 30 分钟，系统必须具备容错与自愈能力。

为应对上述挑战，作者提出了 三大设计原则：

  • P1: 拓扑需贴合 LLM 流量模式，充分利用通信局部性。例如，Tensor Parallelism 产生的集体通信占总流量 50% 以上，且集中于 8–64 个相邻 NPU；而数据并行带来的长距离通信不足 2%。因此，需要分层、局部化的网络架构。
  • P2: 计算与通信需拓扑感知，避免拥塞与带宽浪费；
  • P3: 系统需具备自愈能力，实现链路/节点级故障的快速恢复。

为满足 R1–R4 并遵循 P1–P3，作者提出 UB-Mesh 架构：

  • 采用 分层局部化的 nD-FullMesh 拓扑，优先短程直连，减少交换机与光模块依赖；
  • 实现 分层带宽分配，满足不同范围的通信需求；
  • 引入 全路径路由 (APR) 与拓扑感知容错机制，结合 64+1 高可用设计，提升性能与可靠性。

实验结果表明：

  • UB-Mesh 相比非阻塞 Clos 架构，节省 98% 高端口交换机与 93% 光模块；
  • 在性能损失不足 7% 的情况下，系统级成本效率提升 2.04 倍，可用性提升 7.2%；
  • 在多种 LLM 训练任务中保持 95% 以上的线性扩展性。

2. Background & Motivation

2.1 通信墙问题 (Communication Wall)

  • LLM 训练规模空前，需要成千上万的 NPU/GPU。
  • 每次训练迭代包含前向、反向和参数更新，大量依赖 跨 NPU 的数据交换。随着训练系统的规模增大，数据通信开销成为系统中最昂贵的部分。
  • 随着集群规模扩大，通信成本成为主要瓶颈，即所谓的“通信墙”。

2.2 数据流量的局部性 (Locality of Data Traffic)

  • LLM 训练涉及多种并行方式：
    • Tensor Parallelism (TP)、Sequence Parallelism (SP)、Expert Parallelism (EP)、Pipeline Parallelism (PP)、Data Parallelism (DP)。
  • 不同并行策略产生的数据量差异显著（作者在内部 MoE-2T 模型 上的分析）：
    • TP + SP 占总流量约 97%；
    • 其余方式 (EP/PP/DP) 总计不足 2%。
  • 说明 LLM 训练流量具有 强局部性与层次性，网络设计必须优先支持短距离高带宽通信。

2.3 现有数据中心架构的局限性

  • Clos：性能高，但对称互连成本过高（大量高性能交换机与光模块）。
  • 3D Torus：依赖直连，成本较低，但带宽不足，难支持复杂 All-to-All 通信。
  • DragonFly：比 Clos 便宜，但仍需全带宽交换，短程通信也需经过交换结构，对 LLM 特征流量表现不佳。
  • Fugaku Tofu (6D Torus)：适合 HPC，但与 3D Torus 类似，难以承载 LLM 的通信模式。

3. UB-Mesh 架构 (Architecture)

3.1 nD-FullMesh 拓扑

  • 核心思想：优先使用电缆直连在各层次构建全互连域，逐层递归构造 full-mesh，形成 分层局部化 的拓扑。
  • 构建方式：
    • 1D-FullMesh：板内 NPU 全互连；
    • 2D-FullMesh：机架内相邻 1D 域互连；
    • 3D-FullMesh：跨机架互连；
    • 4D/更高维：跨机房/楼层/数据中心互连。
  • 优势：
    • 充分利用数据局部性与流量模式：nD-FullMesh 在每个网络维度上形成紧耦合的直连域，能够逐层提供超配带宽，从而充分利用 LLM 训练中固有的数据局部性和由密到疏的流量分布特性。还可以灵活调整每个维度的节点带宽分配；
    • 利用数据局部性：大部分通信（通常为 TP 或 SP）在 0–2 跳内完成；
    • 减少交换机/光模块依赖：短距离被动电缆约占总电缆使用量的 86.7%；
    • 支持复杂集体通信：相比 Torus，FullMesh 更适合集体操作（如 All-to-All）。

3.2 UB-Mesh 的基本构件

  • 硬件模块：
    • NPU：AI 计算核心，配备 2×UB 控制器 (72 lanes)；
    • CPU：主机处理，配备 1×UB 控制器 (32 lanes)；
    • LRS (低端口数交换机)：轻量化，主要做机架内/相邻聚合；
    • HRS (高端口数交换机)：大规模互连，Pod 级聚合。

  • Unified Bus (UB) 技术：
    • 单一互连机制，替代 PCIe / NVLINK / IB / RoCE 的混合方案；
    • 优势：
      1. I/O 灵活分配（近端粗带宽 / 远端稀疏带宽）；
      2. 硬件资源池化（CPU、NPU、DDR、NICs 可灵活绑定）；
      3. 系统优化简化（协议统一，减少驱动/库的转换开销）。

3.3 架构实现 (UB-Mesh-Pod)

通过将表 3 中列出的硬件构件组合在一起，并使用统一总线（UB）进行互连，文章提出了 UB-Mesh 的整体架构设计。如图 7-(a) 所示，文章按照所提出的 nD-FullMesh 拓扑实现了一个 UB-Mesh-Pod。具体来说，在该架构中，在每个机架内部构建一个 2D-FullMesh，并将其扩展为跨机架的另一个 2D-FullMesh，最终形成 4D-FullMesh。在本代系统中，选择不将 UB-Mesh 扩展到 5D 或更高维度，以在成本效率与灵活性之间取得工程上的平衡；在 4D-FullMesh 之外，仍采用 Clos 拓扑。

  • 机架级 (Rack)：
    • 机架的核心由 8 块 NPU 板 组成，每块板上包含 8 个 NPU（见图 8-(a)），因此每个机架共有 64 个 NPU。这些 NPU 在机架内通过直连形成 2D-FullMesh 网络，确保 NPU 之间的高带宽通信；
    • 由于 UB I/O 控制器具备路由功能，因此每个 NPU 同时也可作为路由器，实现间接路由；
    • 除 NPU 外，机架还包含独立的 CPU 板（见图 8-(c)）；
    • 机架中配置了多个背板交换机，负责机架内与跨机架的连接。这些交换机采用低端口数设计（LRS），以降低成本，同时确保设备之间的无阻塞通信。整体上，这些背板交换机可导出 4 路 UB x256 I/O。
  • 64+1 高可用设计：
    • 系统在每个机架的 64 个常规 NPU 外，额外配备 1 个备用 NPU（见图 7-(b) 和图 8-(b)）；
    • 该备用 NPU 通过 LRS 与 64 个常规 NPU 相连，当某个 NPU 故障时，管理系统会激活备用 NPU 来接替其工作；
    • 虽增加一跳延迟，但对 LLM 训练整体性能影响极小。
  • Pod 级 (UB-Mesh-Pod)：
    • 在机架内，文章将 64 个 NPU 组织为 2D-FullMesh。跨机架时再实现一个 2D-FullMesh，将机架组织成整体 4D-FullMesh 架构，即 UB-Mesh-Pod。
    • 16 个机架 → 组成 4D-FullMesh（共 1024 NPUs）；
    • 在该配置中，每条链路均为 UB x128 I/O。之所以选择“4 个机架一组、双维互连”，是基于有源电缆的物理可达性约束，这是工程上的最佳平衡点。
  • SuperPod / DCN 扩展：
    • 多个 UB-Mesh-Pod 组成 SuperPod；
    • Pod 之间采用 Clos 拓扑（保证租户隔离和灵活分配）；
    • 通过 UB 交换机或通过位于 CPU 板上的 NIC 接入大规模 DCN，可扩展至 10 万 NPU 级别。

4. 架构实现 Enabling The Architecture

  • 支持混合拓扑：路由系统必须满足 UB-Mesh 的 4D-FullMesh + Clos 混合拓扑带来的需求。

  • 高效转发：由于每个 NPU 也是一个路由器，整个系统包含大量 NPU。为了节省 NPU 硬件资源，路由系统必须高效处理路由表查找和转发操作。

  • 支持非最短路径：在 nD-FullMesh 拓扑中，两端节点之间存在多条不同距离的路径。系统应能够启用非最短路径，以最大化网络带宽利用率。

  • 快速故障恢复：为确保可靠性和可用性，路由系统必须能够快速从故障中恢复，降低对训练过程的影响。

  • 无死锁：整个网络必须在无死锁风险下运行，保证数据流动顺畅不中断。

提出了 全路径路由（All-Path Routing, APR） 技术和 基于直接通知的快速故障恢复 技术，以满足这些要求

4.1 全路径路由（All-Path Routing, APR）

UB-Mesh 的全互连网络架构在任意两个 NPU 之间提供多条路径。传统的路由策略（如最短路径优先 SPF，见图 10-(a)）往往导致网络带宽利用不足，并且容易受到链路故障的影响。为提升网络效率与鲁棒性，文章提出了 全路径路由（APR） 机制。

如图 10-(b) 所示，APR 利用源节点与目的节点之间所有可用路径。这种灵活性允许动态切换路径，以应对故障或拥塞，从而提高网络健壮性。为了高效实现这一目标，APR 依赖以下三种机制：

  • 源路由（Source Routing, SR）机制：实现自适应路由。

    • 为了充分利用 APR 提供的多路径，一个可行的方式是采用源路由机制。如图 11 所示，在原始报文头中添加一个 8 字节的 SR 头部，其中包含转发指令。每个路由器使用一个 4-bit 指针（ptr） 来指示 12-bit 位图字段中的偏移位置。第 i 位的值决定了第 i 跳的转发方式（bit=1 表示该跳采用 SR 转发，bit=0 表示传统转发）。在 SR 转发情况下，位图字段还用于定位六个指令字段之一，指示如何转发该数据包。SR 信息高度压缩，因此开销较低。

  • 结构化寻址与线性表查找：降低路由表查找和转发开销。

    • APR 利用了 UB-Mesh 拓扑的结构化特点，采用了 结构化寻址与线性表查找 机制。

  • 拓扑感知的无死锁流控（TFC）：保证全路径路由下的死锁规避。

    • 该算法在仅使用 2 个虚拟信道的情况下，就能实现无死锁的全路径路由；TFC 通过 信道依赖图（CDG） 对死锁进行建模，并将拓扑划分为子图；在子图内部，统一应用拓扑导向规则与虚拟信道约束；算法采用 跨维度环路消解原则 将子图分解为单 VL 的无环子集，再通过 同维度环路消解原则 来计算排列和笛卡尔积，生成所有路径组合与 VL 映射。

4.2 基于直接通知的快速故障恢复（Fast Fault Recovery via Direct Notification）

当网络链路发生故障时，传统路由系统通常采用逐跳通知机制。如图 12 左侧所示，当链路 1-3 出现故障时，该信息由节点 1 和 3 逐跳传播，通常会消耗较长延迟。

在 UB-Mesh 中，由于每个节点都有确定的一组通信目标，文章可以通过直接通知这些节点来加速路由收敛。如图 12 右侧所示，当链路 1-3 出现故障时，该信息会根据预先计算的路由关系直接发送到节点 6。凭借这种 拓扑感知的直接通知，控制平面的开销大幅减少，从而显著加快了故障恢复。

5. 最大化性能（Maximizing The Performance）

本章节主要针对训练任务的分布以及集合通信的性能进行优化。文章提出了若干拓扑感知的优化策略，以提升性能表现。

5.1 拓扑感知的集合通信（Topology-Aware Collective Communication）

All-Reduce 与 All-to-All 两个典型操作为例。

  • All-Reduce：
    文章提出了一种多环（Multi-Ring）算法，以实现高效的 All-Reduce。首先，将网络拓扑抽象为统一模型，考虑节点数量、节点互连关系、链路带宽和延迟等因素；接着，将集体通信与路径映射结合，采用逻辑多环算法，保证路径独占、避免冲突；如图 13-(a) 所示，原始路径为默认映射。APR 机制可利用这些路径之外的 空闲链路 来增强带宽；最终，如图 13-(b) 所示，文章通过在多条路径间优化流量划分，缓解瓶颈，并最大化 APR 所带来的额外带宽利用。

  • All-to-All：
    文章考虑两类代表性用例：

    • 通用 All-to-All：如图 14-(a)，当 UB-Mesh 中的源节点（示例采用 2D mesh，方法可推广至更高维）向不同目标节点发送不同数据时，文章采用 多路径 All2All 优化。具体而言，每个数据元素（向量或张量）被分为两部分，分别通过 X-FullMesh 与 Y-FullMesh 并行传输，并在最多 1 跳内到达目标。该策略保证了 UB-Mesh nD-FullMesh 架构下的高带宽利用率。
    • 广播 + 规约（Broadcast + Reduce）：对于涉及 token 分发与专家数据收集的 All-to-All 操作，其语义等价于多次广播与规约操作的叠加。如图 14-(b)(c)，文章采用 分层广播/规约 策略，以节省带宽使用，并充分利用 UB-Mesh 的分层拓扑。

5.2 拓扑感知的并行化（Topology-Aware Parallelization）

提出 拓扑感知并行化机制，用于优化 LLM 训练中的模型与数据划分。其流程如图 15 所示：

  • 步骤 1：生成可行的并行化配置，并将其映射到 UB-Mesh 架构；

  • 步骤 2：利用 拓扑感知通信开销模型 对通信成本进行评估；

  • 步骤 3：通过迭代最小化通信开销，找到最优配置。

构建合适的搜索空间，以平衡效率与性能。

  • 采用基于优先级的启发式剪枝：TP 与 SP（或 CP） → 通信量大，优先映射到高带宽域；PP 与 DP → 用于梯度更新，优先级最低；对于 MoE 模型需要的 EP，我们强制 SP×DP 为 EP 的整数倍。
  • 确保开销模型足够准确: 精确建模 APR 与拓扑感知集体通信在 UB-Mesh 拓扑下的行为；并利用高精度的内部仿真平台来校准该模型。

6. 性能评估

文章分为机架内和机架间两个层次，进行比较。探索 UB-Mesh 的架构设计空间，并分析其相较基线 Clos 架构的优势。

6.1 实验设置

  • 基准负载：基准工作负载包括稠密模型和稀疏模型。如LLAMA-70B 与 GPT3-175B 是稠密模型，而 GPT4-2T 是采用 MoE 技术的稀疏模型。
  • 仿真平台：自研的仿真平台，该平台与实际 PoC 硬件保持一致，用于评估集群规模下的 LLM 训练性能。

6.2 机架内架构探索（Intra-Rack Architecture Exploration）

架构对比（见 Fig. 16）：

  • (a) 2D-FM：

    • UB-Mesh 的设计，64 NPUs 电缆直连，形成 2D 全互连拓扑 (X-FM + Y-FM)。
    • 成本低，使用 48 LRS 聚合跨机架带宽与 CPU-NPU 通信。

  • (b) 1D-FM-A：

    • 保留板内 1D X-FullMesh（8 NPU 直连）；
    • 跨板通信由 36 LRS 完成；
    • 每个 NPU：UB x16 → LRS，另 UB x16 → HRS（跨机架通信）。

  • (c) 1D-FM-B：

    • 用 HRS 替代部分 LRS；
    • 背板中 4 LRS 专用于 NPU-CPU 通信；
    • 跨板 NPU 通信由 8 个交换机完成；
    • 每个 NPU 获得 UB x32 的跨机架带宽。

  • (d) Clos：

    • 所有 64 NPUs 端口接入 72 LRS，形成对称 Clos；
    • 提供最高灵活性，但交换机资源消耗极大。

性能比较（见 Fig. 17）：

  • 2D-FM vs Clos：性能为 93.2%–95.9%；
  • 1D-FM-A：比 2D-FM 在 LLAMA2-70B 上提升 2.44%；其他大模型提升 <1.6%；
  • 1D-FM-B：比 2D-FM 提升 >3%，但整体提升仍有限；
  • 序列长度分析：
    • 8K–32K：2D-FM = 95.5%，略低于 1D-FM-A (98.1%)、1D-FM-B (99.2%)；
    • 64K–10M：2D-FM = 95.0%。

✅ 结论：与 Clos 相比，2D-FM 在显著降低硬件成本的情况下，性能差距控制在 7% 以内。

6.3 机架间架构探索（Inter-Rack Architecture Exploration）

架构对比（见 Fig. 18）：

  • (a) 2D-FM：

    • 16 个机架在水平/垂直方向直接互连，形成 2D-FM；
    • 所有机架也连接到 HRS，用于跨 Pod 互连；
    • 支持三种路由策略：
      • Shortest：仅最短路径；
      • Detour：利用 APR，使用更多路径；
      • Borrow：通过交换机“借用”带宽。

  • (b) Clos：

    • 所有机架接入 HRS，无直连；
    • 消耗交换机更多，但提供最高带宽与灵活性。

性能比较（见 Fig. 19）：

  • GPT3-175B：对跨机架通信不敏感；
  • GPT4-2T：
    • Shortest 策略下，2D-FM 比 Clos 低 0.73%；
    • Detour+Borrow 策略下，差距缩小到 0.46%；
  • 总体：2D-FM ≈ Clos，性能几乎相当。

跨机架带宽探索（见 Fig. 20-(b)）：

  • 评估带宽：UB x4 / x8 / x16 / x32 per NPU；
  • 8K–32K 序列：最佳带宽 = UB x16；
    • 从 UB x8 → x16，仅提升 0.44%；
  • 64K–10M 序列：最佳带宽 = UB x32；
    • 从 UB x16 → x32，性能提升 1.85%；

➡️ 长序列下 TP/SP 流量不可避免跨机架，高带宽显著降低通信延迟 → 提升性能。

默认设置：

  • UB-Mesh 默认为每个 NPU 配置 UB x16 I/O 用于跨机架通信；
  • 可调整机架内/机架间带宽比例，以适配不同 LLM 训练任务需求。

6.4 成本效率比较（Cost-Efficiency Comparison）

  • 成本模型：

    • TCO = CapEx + OpEx
    • 成本效率定义为：
      [
      \text{Cost-efficiency} = \frac{\text{Average Performance}}{\text{CapEx + OpEx}}
      ]

  • CapEx 成本（NPU、CPU、LRS、HRS、电缆、模块）：

    • UB-Mesh (4D-FM+Clos) 相比：
      • 2D-FM+x16 Clos：降低 1.18×
      • 1D-FM+x16 Clos：降低 1.26×
      • x64T Clos：降低 1.65×
      • 全 Clos：降低 2.46×
    • 网络基础设施成本占比：从 67% → 20%
    • 节省：98% 高端口交换机、93% 光模块

  • OpEx 成本：

    • 主要为电费与维护费
    • UB-Mesh 相比 Clos 节省 约 35%
    • 在 AI 系统中，OpEx ≈ TCO 的 30%

  • 最终结果：

    • UB-Mesh 成本效率 = 2.04× Clos

6.5 线性扩展性评估（Linearity Evaluation）

  • 线性定义：
    [
    \text{Linearity} = \frac{\text{Per-NPU Perf. on Target Scale}}{\text{Per-NPU Perf. on Base Scale}} \times 100%
    ]

  • 基线规模：

    • LLAMA2-70B：128 NPUs
    • GPT3-175B：512 NPUs
    • Dense-1T、GPT4-2T：1K NPUs

  • 结果：

    • 在 1×–32× 规模下：所有任务线性扩展性均 >100%
      • 因为扩展规模解锁更多高带宽域 + 更优并行策略
    • 在 64×（64K NPUs） 下：
      • GPT4-2T、Dense-1T 线性扩展性下降，但仍 >95%

6.6 网络可靠性分析（Network Reliability Analysis）

  • 公式：
    [
    \text{Availability} = \frac{MTBF}{MTBF + MTTR} \times 100%
    ]

  • 结果：

    • UB-Mesh 使用直连电缆 (E-Cables) → 大幅降低年化故障率 (AFR)
    • MTBF：
      • Clos (8K NPUs)：13.8 小时
      • UB-Mesh：98.5 小时 (7.14× 提升)
    • 可用性：
      • Clos：91.6%
      • UB-Mesh：98.8% (+7.2%)

  • 优化措施：

    • 内部监控工具：
      • 10 分钟内定位故障
      • 3 分钟内触发任务迁移
    • MTTR 优化后，UB-Mesh 可用性提升至 99.78%
  • 成本效率：UB-Mesh = 2.04× Clos
  • 可用性：提升 7.2%
  • 线性扩展性：95%+，适用于多种 LLM 训练任务
  • 性能损失始终 <7%

7. Discussion

  • 集体通信协处理器（CCU）：

    • 集成于 UB I/O 控制器
    • 负责卸载集体通信任务：
      • 执行指令、读写 HBM、发起 NPU 间传输
      • 在片上 SRAM 完成数据规约
    • 优势：
      • 避免冗余拷贝
      • 减少 HBM 带宽压力
      • 保持规约顺序确定性
      • 与计算核心协同，实现 计算–通信重叠

  • 对大规模专家模型的支持：

    • Massive-Expert MoE → 需要大规模、细粒度 All-to-All
    • UB-Mesh 优势：
      • 多路径 + 分层 All-to-All 优化（Sec. 5.1）
      • UB Load/Store 传输
      • CCU 卸载 All-to-All，减少核心消耗

第 8 章 结论（Conclusion）

本文提出 UB-Mesh —— 一种为下一代 LLM 训练设计的新型数据中心网络架构。

  • 拓扑：nD-FullMesh → 最大限度减少交换机与光模块
  • 匹配性：适应 LLM 的分层局部化流量模式
  • 综合成果：
    • 性能 ≈ Clos
    • 成本效率 = 2.04×
    • 可用性 = +7.2%
    • 线性扩展性 = 95%+
