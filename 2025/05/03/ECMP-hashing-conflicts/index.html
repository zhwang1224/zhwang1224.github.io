ECMP 是什么

ECMP（Equal-Cost Multi-Path，等价多路径路由）是传统数据中心网络中最常用的负载均衡机制。当源和目的之间存在多条代价相同的路径时，交换机需要把流量分摊到这些路径上。

ECMP 的核心做法是哈希:交换机对每个数据包的五元组（源IP、目的IP、源端口、目的端口、协议号计算哈希值,再用哈希结果对可用路径数取模,决定这个流走哪条路。
关键特性:

同一条流的所有包走同一条路径(因为五元组不变,哈希结果不变),这保证了不会乱序。
哈希是无状态的,交换机不需要记录每条流的状态,因此可扩展性好。

ECMP 的隐含假设

ECMP 能均匀分摊流量,依赖一个统计假设:网络中存在大量的流。

当流的数量足够多时,即使哈希函数把某几条流碰撞到了同一条路径,从统计平均看,各条路径上的流数量和流量仍会趋于均衡——这就是”大数定律”在起作用。文中描述传统云计算正好满足这个前提:它产生数百万条流,每条流连续且低利用率(通常低于网卡容量的 20%),网络呈现高熵(high entropy)特征,ECMP 工作良好。

什么是 Hashing Conflict(哈希冲突)

哈希冲突指的是:两条或多条不同的流,经过哈希计算后被映射到了同一条出口路径上。在流数量很多时,这种冲突无关紧要;但在流数量很少时,一次冲突就可能让某条路径过载、另一些路径闲置。

LLM 训练的流量模式呈现如下特征：

  1. 流数量极少(低熵)
    传统云计算每个实例产生数十万连接,而 LLM 训练中每个 GPU 只用几十到几百条连接。流的总数从”百万级”塌缩到”几十级”。
  2. 突发性大象流(elephant flow)
    LLM 训练的流量来自梯度同步(AllReduce 等集合通信),呈周期性突发,瞬间就能打满网卡 400Gbps 的全部容量,持续几秒到几十秒。每条流都是”大象流”,数据量巨大。

当少数几条大象流经过 ECMP 哈希时,大数定律失效了。如果两条 400Gbps 的大象流被哈希到同一条物理链路上,这条链路立刻过载丢包,而其他等价路径却完全空闲。这种流量分布不均直接拖慢整个训练。

而传统数据中心是 3 层 Clos 架构(ToR → Aggregation → Core)。一条大象流跨 Pod 通信时,要在三层各做一次哈希,共计三次哈希。问题在于:每一层哈希的输入(五元组)是完全相同的。

如果各层交换机使用相似的哈希算法,相同的输入就倾向于产生相关联的、非独立的输出。本来期望三层哈希像三次独立掷骰子那样把流量充分打散,但实际上由于输入不变、算法相关,流量会被”极化”——系统性地、反复地汇聚到某些路径上,而非随机分散。

级联哈希(cascading hashing)会导致比单层更严重的负载不均。文中观察到大量负载不均案例,尤其集中在需要穿越全部三层的跨 Pod 通信场景。

HPN

  1. 用 2 层架构替代 3 层
    HPN 用一个 2 层双平面架构就容纳 15K GPU(传统需要 3 层 Clos)。层数减少,哈希次数随之减少,从源头上降低了级化的机会。

  2. Rail-optimized + 单芯片大交换机
    靠 51.2Tbps 单芯片交换机和轨道优化网络,1024 个 GPU 装进一个 segment(单层网络)。96.3% 的训练任务规模小于 1K GPU,因此能完全放进一个 segment,根本不需要跨层哈希。

  3. Dual-plane(双平面)消除 Aggregation 层极化
    这是最直接针对哈希极化的设计。把每个 dual-ToR 中的两个 ToR 分入两个物理隔离的平面。
