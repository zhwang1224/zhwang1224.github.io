Micro high radix router

总结

一、早期设计理念与Hop Count最小化

早期的互连网络（如Butterfly和Hypercube）追求最小跳数（hop count），因为这有助于降低延迟。在当时的封装条件下，低radix（端口数少）的路由器延迟更低。

二、现代网络延迟模型

随着技术演进，考虑的因素更多，网络延迟 $T$ 可以分为两部分：

  • 头延迟 $T_h = H \cdot t_r$：跳数 × 每跳延迟

  • 串行化延迟 $T_s = L / b$：数据包长度 ÷ 通道带宽
    总延迟为 $T = H t_r + L/b$。

其中：

  • $H \approx 2 \log_k N$：hop数随radix增大而减少

  • $b = B / 2k$：每个通道的带宽随radix增大而减少

将这些代入，得到总延迟表达式：

$$T = 2 t_r \log_k N + \frac{2kL}{B}$$

三、最优radix的推导

对上式对 $k$ 求导，得到使延迟最小的radix：

$$k \log_2 k = \frac{B t_r \log N}{L}$$

这个比值称为路由器的纵横比（Aspect Ratio），记为 $A = \frac{B t_r \log N}{L}$。
含义如下：

  • $A$ 高 → 更优的是“高而窄”的路由器（端口多、每端口带宽小）

  • $A$ 低 → 更优的是“短而宽”的路由器（端口少、带宽大）

四、技术进步推动radix上升

随着技术进步，路由器总带宽 $B$ 增大，处理延迟 $t_r$ 减少，纵横比 $A$ 增大，从而推动最优radix不断增大：

  • 2003年：Aspect Ratio = 554，最优radix ≈ 40

  • 2010年：Aspect Ratio = 2978，最优radix ≈ 127

五、成本和功耗随radix上升而下降

  • 成本：主要由总带宽决定，与跳数成正比。radix增大 → hop数减少 → 成本降低

  • 功耗：与路由器数量成正比。radix增大 → 路由器节点数减少 → 功耗下降

    • 注意：虽然高radix增加仲裁复杂度，但在总功耗中影响极小

随着技术演进（带宽提高、延迟降低），互连网络的最优拓扑结构也发生了变化。传统小radix设计已不再最优，现代高性能互连网络更倾向于使用高radix、低跳数的拓扑结构，以优化延迟、降低成本和功耗。这种设计趋势已在多代超算中体现出来。

第3节：基线路由器架构（Baseline Router Architecture）

本节介绍一个基础的虚拟通道（VC）路由器设计，它源自低radix路由器的结构。但随着radix（端口数）的提高，这种设计面临分配器（allocator）和连线复杂度急剧上升的问题，导致难以扩展。

后续章节：

第4节：通过分布式分配器和简化VC分配逻辑来缓解复杂性，但仍存在性能瓶颈——头阻塞（head-of-line blocking）。

在一个路由器里，每个输入端口可能有多个虚拟信道（VCs, Virtual Channels），每个 VC 都可能向不同的输出端口发出请求。

  • 如果基数很小，可以用一个集中式仲裁器一次性决定所有输入-输出匹配。

  • 但在高基数（几十到上百个端口）时，这会导致：

    • 布线复杂度极高（每个输入都要连到所有输出仲裁器）。

    • 时延过长（需要在一个时钟周期内处理成百上千个请求）。

    • 芯片面积和功耗过大。

因此，图 6 提出的是一种 分布式、可分解 (separable) 的仲裁机制。整个分配过程分为三阶段：

阶段1：输入仲裁（Input Arbitration）

  • 每个输入控制器内所有准备就绪的虚通道向交叉开关发出请求；

  • 每个输入控制器内部先决出一个优胜虚通道；

  • 该通道将其请求输出端口的编号以二进制形式，通过一组横向请求线发给本地输出仲裁器。

阶段2：本地输出仲裁（Local Output Arbitration）

  • 每个本地输出仲裁器接收来自一组局部输入（如图中 $m = 8$）的请求；

  • 从中选择一个请求转发给全局输出仲裁器。

阶段3：全局输出仲裁（Global Output Arbitration）

  • 全局仲裁器从所有本地仲裁器的输出中，选择一个请求获得访问对应输出端口的权限；

  • 仲裁完成后，授权信号反向传播回输入端的虚通道。

问题：VC 分配比 switch allocation 难，因为：

  • 交换仲裁只关心“输出端口是否空闲”；
  • VC 分配要知道“输出端口的哪条虚通道是否空闲”，这使得 wiring 和逻辑复杂度大幅提升。

理论上，如果每个输入 VC 都能实时监控所有输出 VC 的状态，复杂度会是 v²k²（v=VC 数，k=端口数），不可实现。解决思路：两种可扩展架构:

  • CVA (Crosspoint Virtual Channel Allocation)

    • 思想：在交叉点 (crosspoint) 上保存输出 VC 的状态。

    • 每个 crosspoint 为输出端口的每个 VC 设置一个仲裁器（总共有 k×v 个）。

    • 输入请求发到 crosspoint，本地就能决定是否分配 VC。

    • 优点：实时性好，分配效率高。

    • 缺点：仲裁器数量巨大，硬件开销高。

  • OVA (Output Virtual Channel Allocation)

    • 思想：先完成 switch allocation（即决定了哪个输入能去哪个输出），再检查输出 VC 是否空闲。

    • 好处：不需要在每个 crosspoint 部署仲裁器，硬件简单得多。

    • 代价：由于要等 switch allocation 完成，VC 分配推迟，导致 更深的投机 (speculation)，性能会受损。

第5节：在交换矩阵交叉点增加缓冲区，消除头阻塞问题，但增加的缓冲区面积对芯片代价高。

  • 关键改动：在交叉点 (crosspoint) 增加了缓冲区。

  • 效果：

    • 输入仲裁和输出仲裁解耦。输入端一旦赢得仲裁，就能直接把 flit 放到对应输出的 crosspoint buffer，不用等输出分配完成。

    • 减少投机：传统设计里如果输出仲裁失败，flit 得退回输入队列再等一次仲裁；而现在它已经存在 crosspoint buffer，不会浪费之前的仲裁。

    • VC 分配也简化：因为 flit 已经绑定了输入 VC，到 crosspoint 时只需在本地做一个 v:1 仲裁，再加一个 k:1 的全局仲裁即可。

  • 问题：要避免 crosspoint buffer 溢出。

  • 方法：用 credit-based flow control（基于信用的流控）。

    • 输入端为每个 crosspoint buffer 维护一个“空闲计数”。

    • 发送 flit 时计数减一；flit 出队时返回 credit，计数加一。

  • 挑战：如果每个 crosspoint 单独拉一根返回线，硬件成本太高。

  • 解决方案：所有 crosspoint 共享一条 credit return bus，并在总线上再做分布式仲裁。

  • 结果：仿真发现这种共享总线与“理想情况（即时返回信用）”性能几乎一样，因为 flit 在输入行里本身要走 4 个周期，有足够时间等候仲裁。

上述提到的技术综合起来面积太大，crosspoint buffer 的存储开销随基数 k 呈二次增长，再乘以虚通道数 v，存储面积很快成为瓶颈。一个办法就是：不在 crosspoint 上为每个 VC 分独立 buffer，而是让所有 VC 共享一个 crosspoint buffer。这样存储量减少了 一个因子 v。

但这样也带来了新问题 - VC 分配复杂化。同时需要在共享 buffer 情况下，flit 不能直接等在 crosspoint buffer 里，必须要有 “确认机制”，控制逻辑复杂，还增加了额外的延迟。

第6节：提出分层交换结构（hierarchical switch），在保持良好性能的同时减少芯片面积。 分层交叉开关架构通过灵活调整子交换单元大小，为高radix路由器提供了一种在性能与实现成本之间取得良好平衡的解决方案。未来可以进一步探索如何与路由算法/拓扑结构协同优化，实现系统级最优设计

架构思想

  • 把一个 k 端口 crossbar 拆成若干个 p×p 子交换机 (subswitch)。

  • 每个子交换机有自己的输入缓冲和输出缓冲（per-VC）。整个大 crossbar 就是由这些小 crossbar 拼成。

  • 缓冲区面积缩减：因为只有子交换机边界有 buffer，总存储开销变成 O(v·k²/p)。相比 fully buffered（O（v·k²）），大幅下降。

控制逻辑

  • 本地控制 (local)：每个子交换机只需要处理自己的输入输出情况，仲裁规模小，延迟低。

  • 全局控制 (global)：在子交换机之间协调，确保 flit 能跨越整个 crossbar。

  • VC allocation 被拆成两层：

    • 子交换机内部做 local VC allocation。

    • 子交换机之间做 global VC allocation。

  • 好处是：VC 分配失败时，flit 只会卡在子交换机缓冲，不需要 NACK 回退，也不会死锁。

✅ 总体结论

  • 随着技术的进步，为了更高效地利用芯片带宽，高radix路由器成为趋势；

    • 相比于传统的少端口宽通道结构，高radix采用多端口窄通道，能更有效地降低延迟与成本。

  • 然而，现有路由器微架构无法直接扩展到高radix：

    • 直接扩展基线架构虽然设计简单，但吞吐率不到50%；

    • 全缓冲交叉开关（fully-buffered crossbar）在每个交叉点都设置了虚通道缓冲区，虽能接近100%吞吐，但面积和成本极高。

🌟 作者提出的解决方案：分层交叉开关（Hierarchical Crossbar）

  • 性能接近全缓冲结构，但成本可接受；

  • 相比基线架构，吞吐率提升20–60%；

  • 相比全缓冲结构，芯片面积节省约40%；

  • 分层结构不仅降低了硬件成本，还便于将控制逻辑也按层次结构组织，简化设计和实现。
