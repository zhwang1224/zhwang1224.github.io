大模型推理基础

发表于 2026-07-16 分类于 人工智能系统

大模型推理基础

一、大模型推理

大模型通常经历训练和推理两个阶段。

  • 训练（Training）：模型读取大量样本，通过前向传播得到预测结果，计算损失，再通过反向传播和优化器更新参数。
  • 推理（Inference）：模型参数通常保持不变，系统加载训练完成的权重，对新的输入执行前向计算并生成结果。

可以把训练理解为“形成模型能力”，把推理理解为“在真实请求中使用这种能力”。ChatGPT、智能客服、代码助手、RAG 问答和 Agent 系统对外提供服务时，主要执行的都是推理。

不过，在工程语境中还需要区分 inference 和 serving：

  • inference 更强调模型完成一次前向计算或生成过程；
  • serving 则包括模型加载、请求接入、排队、批处理、调度、并行执行、KV Cache 管理、流式传输、监控和故障恢复等完整服务链路。

因此，一个模型“理论推理很快”，并不代表它在生产服务中的端到端延迟一定很低。客户端网络、服务排队、批处理策略和跨设备通信都可能进入关键路径。

虽然推理不需要保存梯度和优化器状态，也不执行反向传播，但在线服务面对的是另一组挑战：大量用户同时访问、输入和输出长度动态变化、持续增长的 KV Cache、严格的首 Token 延迟，以及吞吐量与单用户延迟之间的权衡。推理并不一定比训练简单，只是优化目标不同。

对比维度 模型训练 模型推理
核心目标 提高模型精度和能力 降低延迟、提高吞吐量和性价比
参数状态 持续更新 通常只读
主要计算 前向传播、反向传播、优化器更新 Prefill 与自回归 Decode
额外状态 激活值、梯度、优化器状态 KV Cache、请求状态、运行时缓冲区
常见并行方式 数据、张量、流水线、专家并行 张量、流水线、专家并行和请求批处理
主要资源压力 计算量和训练状态容量 权重读取、KV Cache、内存带宽和通信

二、从输入文本到输出 Token：一次推理请求的完整过程

一个典型的自回归 LLM 请求可以划分为请求接入、Tokenization、Prefill、Decode、Detokenization 和流式返回等阶段。

其中，Prefill 和 Decode 都运行同一个 Transformer 模型，但二者处理的张量形状、并行度、访存特征和主要性能瓶颈明显不同。这一区别是理解 LLM 推理系统的关键。

1. Tokenization：文本如何成为模型输入

神经网络的核心数值运算主要是矩阵乘法，例如：

[
\mathbf{Y}=\mathbf{X}\mathbf{W}
]

此外还包括向量加法、归一化、激活函数和 Softmax 等操作。模型的计算图不能直接处理“我喜欢机器学习”这样的字符串及其语义，因此需要先通过 Tokenizer 将文本编码为整数形式的 Token ID。

现代 LLM 常使用 BPE、WordPiece、Unigram、SentencePiece 或字节级编码及其变体。它们可能把文本拆分为词、子词、字符、标点或字节片段。例如，输入：

设输入文本经过分词后得到：

[
\mathbf{x}=[x_1,x_2,\ldots,x_S]
]

其中 $S$ 是输入序列长度。每个 $x_i$ 是词表中的整数编号。编号本身只用于索引，并不存在“编号越大、语义越强”这样的关系。

从 Token ID 到词向量

模型内部维护一张可训练的嵌入表：

[
\mathbf{E}\in\mathbb{R}^{|V|\times d_{model}}
]

其中 $|V|$ 是词表大小，$d_{model}$ 是隐藏维度。对于 Token ID $x_i$，嵌入层本质上是从 $\mathbf{E}$ 中取出第 $x_i$ 行：

[
\mathbf{e}_i=\mathbf{E}[x_i]
]

例如，文本“我 喜欢 猫”被编码为：

假设 $d_{model}=4$，嵌入查询可能得到：

[
101\rightarrow[0.2,-0.1,0.7,0.3]
]

[
205\rightarrow[0.8,0.4,-0.2,0.1]
]

[
309\rightarrow[-0.3,0.9,0.5,0.6]
]

这些数值同样只是示意。将三个 Token 的向量按文本顺序排列，可以得到单条序列的初始表示：

[
\mathbf{X}=
\begin{bmatrix}
0.2 & -0.1 & 0.7 & 0.3\
0.8 & 0.4 & -0.2 & 0.1\
-0.3 & 0.9 & 0.5 & 0.6
\end{bmatrix}
\in\mathbb{R}^{3\times4}
]

这里：

  • 每一行对应一个 Token；
  • 每一列对应一个隐藏特征维度；
  • $S=3$，表示序列包含 3 个 Token；
  • $d_{model}=4$，表示每个 Token 用 4 个数表示。

因此，单条文本的隐藏状态形状为：

[
\mathbf{X}\in\mathbb{R}^{S\times d_{model}}
]

实际服务通常一次处理多条请求。加入 batch 维度后，隐藏状态矩阵写为：

[
\mathbf{X}\in\mathbb{R}^{B\times S\times d_{model}}
]

例如，一次处理 $B=2$ 条文本，每条文本经过 Padding 或批处理调度后具有 $S=3$ 个 Token，每个 Token 的隐藏维度为 $d_{model}=4$，那么整体张量形状就是：

[
\mathbf{X}\in\mathbb{R}^{2\times3\times4}
]

“隐藏状态”中的“隐藏”不是指数据被加密，而是指它属于模型内部的中间表示，人通常无法给每个维度赋予稳定、独立的自然语言含义。

位置如何进入模型？

仅有词嵌入还不足以区分 Token 的先后顺序。不同架构会采用不同的位置编码方式：

  • 早期 Transformer 常把绝对位置嵌入直接加到 Token Embedding 上；
  • Llama、Qwen 等许多现代 Decoder-only LLM 使用 RoPE，在注意力计算中对 Q 和 K 注入相对位置信息；
  • 还有模型使用 ALiBi 或其他位置偏置方法。

因此，更一般地说，最初的 $\mathbf{X}^{(0)}$ 包含 Token 的语义表示，而位置信息会根据具体架构在嵌入阶段或注意力阶段被注入。隐藏状态经过每一层 Transformer 后不断更新：

[
\mathbf{X}^{(0)}
\rightarrow\mathbf{X}^{(1)}
\rightarrow\mathbf{X}^{(2)}
\rightarrow\cdots
\rightarrow\mathbf{X}^{(L)}
]

随着层数加深，每个 Token 的表示逐渐融入当前因果上下文中的语法、语义和任务信息。

同一段自然语言在不同 Tokenizer 下可能产生不同数量的 Token。因此，模型上下文上限、Prefill 计算量、KV Cache 容量、推理费用都应以 Token 数而不是字符数衡量。

2. Transformer 单层在计算什么？

为了理解 Prefill、Decode 和 KV Cache，需要先观察一个简化的 Transformer Decoder 层。

2.1 从隐藏状态生成 Q、K 和 V

自注意力机制需要解决的问题是：对于当前位置，前文中哪些位置的信息更加重要？为此，输入隐藏状态会经过三组不同的线性投影，得到 Query、Key 和 Value：

[
\mathbf{Q}=\mathbf{X}\mathbf{W}_Q,\qquad
\mathbf{K}=\mathbf{X}\mathbf{W}_K,\qquad
\mathbf{V}=\mathbf{X}\mathbf{W}_V
]

可以把三者直观理解为：

  • Query（Q）：当前位置正在查找什么信息；
  • Key（K）：当前位置能够与什么查询相匹配；
  • Value（V）：如果当前位置被关注，它实际提供什么内容。

这种解释有助于理解注意力，但 Q、K、V 本质上都是模型训练得到的连续向量，并不是人工定义的数据库查询字段。

2.2 缩放点积注意力

对于第 $h$ 个注意力头，缩放点积注意力为：

[
\operatorname{Attention}(\mathbf{Q}_h,\mathbf{K}_h,\mathbf{V}_h)
=\operatorname{Softmax}
\left(
\frac{\mathbf{Q}_h\mathbf{K}_h^{\mathsf T}}{\sqrt{d_h}}+\mathbf{M}
\right)\mathbf{V}_h
]

其中 $d_h$ 是单个注意力头的维度，$\mathbf{M}$ 是因果掩码。

矩阵 $\mathbf{Q}\mathbf{K}^{\mathsf T}$ 计算每个 Query 与每个 Key 的向量点积。假设序列有 4 个 Token，单头注意力分数矩阵的形状就是 $4\times4$：

[
\mathbf{Q}\mathbf{K}^{\mathsf T}=
\begin{bmatrix}
\mathbf{q}_1\cdot\mathbf{k}_1 & \mathbf{q}_1\cdot\mathbf{k}_2 & \mathbf{q}_1\cdot\mathbf{k}_3 & \mathbf{q}_1\cdot\mathbf{k}_4\
\mathbf{q}_2\cdot\mathbf{k}_1 & \mathbf{q}_2\cdot\mathbf{k}_2 & \mathbf{q}_2\cdot\mathbf{k}_3 & \mathbf{q}_2\cdot\mathbf{k}_4\
\mathbf{q}_3\cdot\mathbf{k}_1 & \mathbf{q}_3\cdot\mathbf{k}_2 & \mathbf{q}_3\cdot\mathbf{k}_3 & \mathbf{q}_3\cdot\mathbf{k}_4\
\mathbf{q}_4\cdot\mathbf{k}_1 & \mathbf{q}_4\cdot\mathbf{k}_2 & \mathbf{q}_4\cdot\mathbf{k}_3 & \mathbf{q}_4\cdot\mathbf{k}_4
\end{bmatrix}
]

其中：

  • 第 $i$ 行描述位置 $i$ 对各历史位置的关注程度；
  • 第 $j$ 列对应位置 $j$ 被其他位置查询时的匹配分数；
  • $\mathbf{q}_i\cdot\mathbf{k}_j$ 是位置 $i$ 的 Query 和位置 $j$ 的 Key 的点积。

如果四个 Token 是“我、喜欢、吃、苹果”，最后一行描述的就是“苹果”这个位置分别与“我、喜欢、吃、苹果”匹配到什么程度。

2.3 为什么需要除以 $\sqrt{d_h}$？

当向量维度 $d_h$ 增大时，点积的数值幅度通常也会增大。如果直接送入 Softmax，分布容易过于尖锐，造成梯度或数值不稳定。因此使用：

[
\frac{\mathbf{Q}\mathbf{K}^{\mathsf T}}{\sqrt{d_h}}
]

对分数进行缩放，使其保持在更稳定的范围。

2.4 因果掩码

自回归模型在训练和 Prefill 时虽然可以并行计算多个位置，但位置 $i$ 不能使用位置 $i+1$ 及之后的信息。对于长度为 4 的序列，加性因果掩码可以写成：

[
\mathbf{M}=
\begin{bmatrix}
0 & -\infty & -\infty & -\infty\
0 & 0 & -\infty & -\infty\
0 & 0 & 0 & -\infty\
0 & 0 & 0 & 0
\end{bmatrix}
]

未来位置的分数加上 $-\infty$ 后，经 Softmax 转换为 0。于是，位置 $t$ 只能关注位置 1 到 $t$，不能提前看到未来 Token。

2.5 Softmax 与 Value 加权求和

注意力分数可以是任意实数。Softmax 将每一行转换为非负且总和为 1 的注意力权重。假设“苹果”对应位置得到的权重为：

[
[0.07,0.10,0.62,0.21]
]

则该位置的注意力输出为：

[
\mathbf{o}{\text{苹果}}
=0.07\mathbf{v}{\text{我}}
+0.10\mathbf{v}{\text{喜欢}}
+0.62\mathbf{v}{\text{吃}}
+0.21\mathbf{v}_{\text{苹果}}
]

因此，缩放点积注意力公式可以拆解为五步：

  1. 用 $\mathbf{Q}\mathbf{K}^{\mathsf T}$ 计算位置之间的匹配分数；
  2. 用 $\sqrt{d_h}$ 缩放数值；
  3. 用因果掩码 $\mathbf{M}$ 屏蔽未来位置；
  4. 用 Softmax 得到归一化注意力权重；
  5. 根据权重对 $\mathbf{V}$ 进行加权求和。

2.6 多头注意力

模型不会只执行一组注意力，而是并行使用多个注意力头。不同注意力头可以在训练中形成不同的表示子空间，有些头可能对局部搭配、长距离依赖、指代、位置或格式信息更加敏感。不过，这种分工是模型训练后形成的统计现象，不应把每个头理解成具有固定、人工指定的功能。

假设：

[
d_{model}=4096,\qquad H_q=32
]

在标准 MHA 且各头维度相等时：

[
d_h=\frac{d_{model}}{H_q}
=\frac{4096}{32}=128
]

各个头分别完成注意力计算，然后拼接并通过输出投影：

[
\operatorname{MHA}(\mathbf{X})
=\operatorname{Concat}
(\mathbf{O}1,\ldots,\mathbf{O}{H_q})\mathbf{W}_O
]

对于 GQA 和 MQA，Query Head 数量与 KV Head 数量并不相等，但多个 Query Head 会按组共享 K/V；后文将在 KV Cache 部分进一步说明。

2.7 残差连接、归一化与 MLP

注意力产生的新信息不会直接覆盖原隐藏状态，而是通过残差连接与输入相加。许多现代 LLM 使用 Pre-Norm 结构，一个简化的 Transformer 层可以写成：

[
\widetilde{\mathbf{X}}
=\mathbf{X}
+\operatorname{MHA}(\operatorname{Norm}(\mathbf{X}))
]

[
\mathbf{Y}
=\widetilde{\mathbf{X}}
+\operatorname{MLP}(\operatorname{Norm}(\widetilde{\mathbf{X}}))
]

残差连接为深层网络提供更直接的信息和梯度路径；LayerNorm 或 RMSNorm 等归一化方法则帮助控制数值尺度、稳定深层网络计算。

MLP 主要对每个 Token 的隐藏向量独立进行非线性变换。以常见的 SwiGLU 为例，可以简化表示为：

[
\operatorname{MLP}(\mathbf{x})
=\left[
\operatorname{SiLU}(\mathbf{x}\mathbf{W}{gate})
\odot(\mathbf{x}\mathbf{W}{up})
\right]\mathbf{W}_{down}
]

其中 $\odot$ 表示逐元素乘法。注意力负责在 Token 之间混合信息，MLP 则在每个 Token 内部进行特征变换；二者共同构成 Transformer 层的主要计算。

2.8 从最终隐藏状态到下一个 Token

所有 Transformer 层执行结束后，最终隐藏状态经过 LM Head 投影到词表维度，得到 logits：

[
\mathbf{z}_t=\mathbf{h}t\mathbf{W}{vocab}
]

其中 $\mathbf{z}_t\in\mathbb{R}^{|V|}$，每一个元素对应词表中一个 Token 的未归一化分数。经过 Softmax 后得到概率分布：

[
P(x_{t+1}=v\mid x_{\leq t})
=\operatorname{Softmax}(\mathbf{z}_t)_v
]

系统再通过 greedy decoding、temperature、top-k、top-p 或其他采样策略选择下一个 Token。被选中的 Token 会加入上下文，随后进入下一轮 Decode。

3. Prefill：并行理解完整输入上下文

Prefill 也称 prompt processing 或 context encoding。它接收全部 $S$ 个输入 Token，并在每一层同时计算这些 Token 的 Q、K、V、Attention 和 MLP。

在单层自注意力中，Prefill 的 Q、K、V 都覆盖整个序列。忽略常数项后：

  • Q/K/V 投影和 MLP 的计算量大致随 $S$ 线性增长；
  • 注意力分数 $\mathbf{Q}\mathbf{K}^{\mathsf T}$ 的计算和中间逻辑规模随 $S^2$ 增长；
  • 每层都会生成并保存全部输入 Token 的 K 和 V，作为后续 Decode 的初始 KV Cache。

可将注意力部分的时间复杂度简化表示为：

[
O(B\times H_q\times S^2\times d_h)
]

其中 $H_q$ 是 Query Head 数量。严格来说，完整 Transformer 的主要线性层还包含与 $d_{model}^2$ 相关的计算，因此不能只用 $O(S^2)$ 概括整个模型成本。

Prefill 的重要特点是：序列中大量 Token 可以并行处理，形成尺寸较大的矩阵乘法，通常更容易提高 Tensor Core 或矩阵计算阵列的利用率。因此，中短上下文下的 Prefill 经常表现出较强的 compute-bound 特征。

但“Prefill 一定是计算受限”并不严谨。随着上下文增长，注意力矩阵的计算和数据移动迅速增加，HBM 带宽、片上缓存容量以及 Attention Kernel 的 I/O 效率也可能成为瓶颈。FlashAttention 等技术正是通过分块和算子融合减少 HBM 读写，而不是改变注意力的数学结果。

Prefill 完成后，最后一个输入位置的隐藏状态产生首个输出 Token 的 logits。因此，Prefill 性能会直接影响 TTFT。输入越长、系统排队越严重，用户等待首 Token 的时间通常越长。

4. Decode：为什么只能逐 Token 生成？

自回归语言模型的联合概率可以分解为：

[
P(x_{1:T})=\prod_{t=1}^{T}P(x_t\mid x_{<t})
]

第 $t+1$ 个 Token 的概率依赖包括第 $t$ 个 Token 在内的全部历史。因此，对于同一条序列，后一个输出 Token 必须等待前一个 Token 被选出，输出长度维度上存在天然的串行依赖。

每个 Decode Step 的过程是：

  1. 将上一轮选出的 Token 送入模型；
  2. 在每一层只计算这个新 Token 的 Q、K 和 V；
  3. 用新 Query 与 KV Cache 中全部历史 Key 计算注意力分数；
  4. 对历史 Value 加权求和；
  5. 把新 K、V 追加到缓存；
  6. 通过 MLP、LM Head 和采样得到下一个 Token；
  7. 重复执行，直至生成 EOS、命中停止条件或达到长度上限。

假设当前上下文长度为 $S_t$，单个请求在某层 Decode 时的主要张量形状可近似写为：

[
\mathbf{Q}_t\in\mathbb{R}^{B\times H_q\times1\times d_h}
]

[
\mathbf{K}{cache},\mathbf{V}{cache}
\in\mathbb{R}^{B\times H_{kv}\times S_t\times d_h}
]

新 Query 需要与历史 Key 做长度为 $S_t$ 的注意力计算。随着输出不断生成，$S_t=S_{input}+S_{generated}$ 持续增大，所以较长回答后半段的单 Token 延迟可能高于前半段。

与 Prefill 相比，Decode 的序列维度通常只有 1，矩阵乘法更“瘦”，单个请求提供的计算并行度有限。然而，模型每生成一个 Token，仍需要执行各层的大规模权重投影。低 batch 场景下，同一批权重的数据复用不足，硬件往往花费大量时间把权重从 HBM 或其他存储层级搬运到计算单元，因此 Decode 通常更偏向 memory-bandwidth-bound。

对 batch size 约为 1 的粗略理想模型，可以写出：

[
\text{Decode Tokens/s}
\lesssim
\frac{\text{Effective Memory Bandwidth}}
{\text{Model Weight Bytes per Step}+\text{KV Traffic}}
]

这不是精确性能公式，但能够解释为什么峰值 FLOPS 很高的设备，单用户输出速度仍可能不高。增大 batch 可以让多个请求复用同一轮权重读取，提高总吞吐量；代价是可能增加排队时间、调度开销和单请求延迟。
